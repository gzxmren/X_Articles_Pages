Hermes 跑通了，然后呢？7 天把它变成工作台
22K
跑通 Hermes 的当天，我没有马上轻松下来。
命令能跑，模型能答，TUI 能打开，hermes --version 也正常。照新手教程的标准，这事已经结束了。可终端一关，我马上又回到老样子，手动刷信息，手动翻聊天记录，手动复制同一段要求，再问 Agent 刚才做到哪了。
这就有点尴尬了。工具已经装好，工作却没有被接住。
后来我才把问题想清楚。Hermes 跑通，只说明它能被调用；它有没有用起来，要看第二天还剩下什么。有没有可恢复的 session，有没有能继续推进的任务，有没有能留给下一轮 Agent 的状态，有没有一套下次还能复用的规则。
所以我没有上来就搭什么 7x24 小时 Agent 团队。那种说法很容易让人兴奋，也很容易第一天就把自己绕晕。我做的事很小，把这一周里反复出现的工作切成几块，逐个放进 Hermes 能接住的位置。
我只看三件事。
它能不能接住真实工作
它能不能在中途出错时恢复
它能不能把今天做过的事留给明天继续用
这 7 天，不是安装教程，也不是功能清单。它更像我给自己补的一份使用顺序。
第 1 天，先做一次冷启动检查。
我现在不太相信“安装成功”这四个字。很多工具安装成功，只代表命令行里没有报错。真的拿来做事，问题才会冒出来，模型可能没配好，session 可能恢复不了，当前目录可能读不明白，任务状态也可能没有地方放。
Hermes 还更麻烦一点。它不是一个单点工具，provider、session、memory、skill、gateway、profile、kanban 几层东西叠在一起。只看版本号，什么都看不出来。
第一天我不急着建一套漂亮流程，也不急着自动化。我只做一次冷启动检查。
hermes --version
hermes status
hermes sessions list
hermes kanban boards list
这四条命令看起来很普通，看的东西却不一样。--version 看命令能不能响，status 看 provider、gateway、cron 这些底座是不是活着，sessions list 看上一轮对话有没有留下可恢复的记录，boards list 看任务状态有没有地方落。
这里不需要一个好看的任务名，也不需要一上来就有完整项目。第一天最怕的是自己骗自己，以为能聊天就等于能接活。聊天窗口里的回答很容易消失，session 和 event 留不下来，第二天还是要从头解释。
所以我会再做一个很小的动作，随便找一个测试任务，写一条 comment，让它进入 running 或 blocked。能看到 task id、comment、event、run、heartbeat，才说明 Hermes 开始有工作台的样子。
第一天只要确认这些痕迹能留下来就够了。版本、状态、session、board、comment、event 都在，后面的记忆、Skill、信息源和多 Profile 才有地方接上。
第 2 天，先写记忆规则，别让它什么都记。
很多人跑通 Hermes 后，很快会问，它为什么还是不记得。
我一开始也这么问。后来发现，麻烦常常不是它记得少，是我乱塞东西。临时想法、半成品判断、一次性报错、过时偏好，全都写进记忆里，后面每次对话都会被旧东西拖住。
后来我给自己定了一个很笨的规矩，先分清楚位置。
SOUL.md 只放长期原则。比如真实优先于顺从，不编造来源，危险操作前确认，发现我的判断有问题要指出。
USER.md 放稳定偏好。比如默认中文，不需要一直赞同我，方向不靠谱要直接提醒。
MEMORY.md 放环境事实。工作区在哪，文章目录怎么放，哪些流程真的跑通过。
AGENTS.md 放项目规矩。某个仓库里测试怎么跑，哪些文件不能碰，改代码后要做什么检查。
这几份文件不是越满越好。它们更像几道栏杆，告诉 Agent 什么可以默认相信，什么必须重新确认。
这版可以直接抄。但我建议抄完第一件事就是删，删到只剩自己的真实情况。
# SOUL.md

你是我的长期工作搭档，不是一次性问答机器人。

【永久原则】

- 真实优先于顺从。不要为了让我舒服而附和我。
- 如果我的判断、方案或事实有问题，直接指出，并说明为什么。
- 不编造来源、链接、数据、文件名和工具结果。不确定就说“不确定”。
- 把事实、推断、建议分开写，不要混在一起装成结论。
- 做危险操作前先说明影响范围，等我确认后再执行。
- 长任务先拆步骤，再推进；卡住时明确说明卡在哪里。
# USER.md

【我的偏好】

- 我默认使用中文。
- 我更重视正确性，不需要你一直赞同我。
- 我喜欢直接、具体、有证据的反馈，不喜欢泛泛鼓励。
- 如果我提出的方向不靠谱，先指出问题，再给更好的替代方案。
- 给建议时请明确推荐项，不要把所有选项说成一样好。
- 我不喜欢夸张营销、空洞口号和没有根据的确定语气。
# MEMORY.md

【固定工作区】

- 我的主工作区是: /Users/你的用户名/workplace
- 文章项目放在: 01-内容生产/文章/
- 每篇文章单独一个目录，包含 00-调研、01-草稿、02-配图、03-发布物、04-发布记录。

【已跑通流程】

- 这里放已经验证过的稳定流程，例如写文、开发、调研、发布、数据分析。
- 每条流程都要能说清楚输入、输出和完成标准。
- 没有跑通过的流程，不要写进“已跑通流程”，先放到“待验证”。
记忆规则别让 Agent 自己编。它不知道我哪里被坑过，也不知道哪些偏好只是今天临时起意。
我现在宁愿少写。每个文件先放 5 条，跑一周再看。能删掉的删掉，比继续往里加更重要。
第 3 天，把重复要求写成 Skill。
第三天我才会碰 Skill。
太早碰 Skill，很容易变成收藏夹。看到一个好 prompt 就封装，看到一个流程就保存，最后攒出一堆差不多的文件，真用的时候反而不知道该叫谁。
我现在会先问三个很土的问题。
它是否重复出现
它有没有明确输入和输出
它有没有验收标准
比如“筛选 X 上值得写的 AI 动态”，这件事就可以写成 Skill。输入是推文或 transcript，输出是候选线索、可写角度、是否需要继续验证。验收也清楚，不能像翻译，不能只转述，要有从业者判断。
“帮我写得高级一点”就不行。太虚，输入不清，最后好不好也说不明白。
我把 Memory 和 Skill 分得很开。Memory 记事实，Skill 记动作。
比如我不喜欢公开正文堆参考链接，这是一条偏好。可“生成发布包时正文不放链接，X Article 如需保留来源就提醒我放到评论区”，这就是流程。
第三天不用做很多事。从最近三次反复说过的话里挑一句，改成这样的格式就够了。
当我给你一批 AI 工具更新时:
- 按 agent / coding / image / video / infra 分类
- 每条保留原始链接和更新时间
- 只挑 5 条值得继续看的
- 每条说明为什么值得写
- 最后给一个不超过 120 字的推荐理由
这套要求我已经手打过太多遍。再靠临场复制，迟早会漏。我把它交给 Hermes，不指望它读心，只是让它每次按同一个格式交活。
第 4 天，接信息源前先定标准。
信息源这件事，我后来变得很谨慎。
一开始最容易犯的错，是觉得源越多越好。今天刷 50 个账号嫌不够，明天又想接 160 个 RSS。看起来很勤奋，实际只是把焦虑变成了自动化流程。
Hermes 可以定时抓取、归纳、推送，但它不知道我今天该看什么，也不知道哪条东西值得写。这个判断不能交给管道。管道只能搬运，标准要先由我写出来。
我先写了一张很粗的筛子。
保留:
- 反直觉观点
- 行业判断
- 真实困惑
- 技术争议

跳过:
- 纯产品发布
- 情绪表达
- 没有可执行动作的宏大判断
如果是 X，我不会一上来追求抓全量。先从固定账号列表或 RSS 开始，看每天出来的东西有没有用。连续几天都挑不出可写内容，通常不是抓得不够，是筛选标准写错了。
YouTube 也是一样。我不把它当娱乐流刷。访谈、官方演示、开发者活动、工具教程，先转成 transcript 或笔记，再让 Hermes 找里面的新信息、争议点和可复现动作。
第四天的要求很小，每天只给我一张候选卡片。
每天只输出 5 条可写线索。 每条包含: 
- 原始来源
- 一句话摘要
- 为什么值得写
- 我能从哪个角度写
- 适合公众号、X，还是只适合收藏
- 是否需要继续验证
这一步救不了写作，但能救选题前那半小时。以前那半小时经常是在浏览器标签页里乱跳，现在至少知道该从哪五条开始看。
第 5 天，把写文流程搬进 Kanban。
到这里，聊天记录已经不够用了。
以前写一篇文章，状态总是散的。选题在聊天里，资料在文件夹里，截图在桌面上，草稿在 Obsidian，发布记录又在另一个窗口。Agent 做到一半断了，下一轮还要重新解释，现在写到哪里，哪张图能用，哪些平台只是存了草稿。
Kanban 对我有用，不是因为它好看，是因为它能把“做到哪了”这件事钉住。
这里我差点写假。文章叫 7 天实战记录，我一开始也想建一个 Day 1 到 Day 7 的 board，每个小标题一张卡。那样截图很好看，结构也整齐，但它不是任务，只是目录。
第 1 天，健康检查 可以当文章小标题，却不一定能当 Kanban 任务。它没有明确交付边界，也没有真实阻塞条件。为了文章硬建一个 7 天 board，只能证明我会敲命令，不能证明它进了生产。
后来我改成按产物切。
确认选题与角度
收件箱新增素材整合
写 human-stance.md
写 sources-ledger.md
写 draft-v1.md
根据反馈改 draft-v2 / draft-v3
准备实操截图
生成发布包
存草稿
写发布复盘
我后来只保留一种标准。卡片完成以后，必须真的多出一个文件，或者让一个文件进入可审核状态。
现在我写 Kanban 任务，会先问四句。
完成后会多出什么文件
谁来确认它算完成
卡住时应该 block 在哪里
下一个任务依赖它什么
答不上来，它大概率只是一个想法，还不是 Kanban 任务。
写文流程也是这样。选题确认是 ready，调研中是 running，缺官方来源就 blocked，正文写完才 done，等人工审稿也应该 blocked。状态比聊天记录可靠，因为状态不会假装自己记得。
Kanban 也不是万能。它只能让状态看得见，不能保证流程安全。真要让它接生产任务，我还会加三条安全带。
风险操作前，先留快照 文件写坏了，要能回滚 长任务跑偏时，要能中途转向
这几件事不酷，但它们会降低心理成本。知道它能停、能回、能改方向，我才愿意把更长的任务交给它。
第 6 天，再尝试多 Profile。
市场上很多 Hermes 文章喜欢写总管、市场总监、产品总监、架构总监、开发总监、测试总监。听起来像一夜之间有了一个团队。
我不建议第一周就这么做。
多 Agent 最大的问题不是创建角色，是边界。谁负责调研，谁负责写作，谁能改文件，谁能调用浏览器，谁能发布，谁能在卡住时 block 任务，谁有权把任务交给下一个人，这些不写清楚，多 Agent 只会互相制造上下文负担。
我更愿意从两个 profile 开始。
researcher: 只负责信息筛选、来源台账、候选角度 writer: 只负责正文、去 AI 味、平台改写
再加人工 gate。
researcher 不能直接写正文 writer 不能自己选题 任何发布动作都必须等人工确认
这套配置看起来保守，用起来很稳。Agent 协作不是角色越多越强，边界越清楚越稳定。
两个 profile 能稳定交接，再加 reviewer。三者能稳定跑，再加 image。正文、图片、发布都稳定了，再谈更大的团队。
如果已经配了多个模型，也可以让主模型负责判断和写作，便宜快的模型负责标题、摘要、压缩、初筛。不过第一周我不会把省模型费用放在前面。角色边界跑顺，比省一点钱重要。
第六天只确认一件事。同一个任务，能不能被不同角色接手，同时不丢状态。
第 7 天，留下复盘，不只留截图。
Agent 工作流很容易变成演示。跑的时候很热闹，截图很多，命令很多，最后什么都没沉淀。下一次做同类任务，又从头开始。
我现在要求每个流程结束后，至少留下三样东西。
这次最终产物在哪
哪些步骤下次可以复用
哪些地方必须人工确认
比如这篇文章，最后应该留下文章项目目录、一组现场实操图、human-stance.md、sources-ledger.md、草稿、写文任务链，还有一份复盘，记录哪些状态真实推进过，哪些只是演示。
如果已经在用 Obsidian 或自己的知识库，我还会再走一步，把这次产物收进去。
raw/        原始资料，不改写
concepts/   沉淀出来的概念
moc/        主题地图
drafts/     可发布草稿
log.md      这次更新了什么
这个动作看起来慢，但很值。Agent 最怕的不是不会归纳，是归纳完没有地方去。文章发完以后，截图躺在文件夹里，判断散在聊天里，下次写同类主题又从头找。知识库的作用不是收藏，是让下一次 Hermes 有地方接着长。
如果只留下“今天玩了一下 Hermes，很强”，那就还是玩具。
如果留下的是“下一次写 Hermes / Agent / 工作流文章，可以从这套任务链继续复制”，它才开始变成生产资产。
发布后我会补一份 publish-log。
这篇文章发到哪些平台
哪些标题有效
哪张图被点击
哪一步最费时间
下次应该提前准备什么
这次有哪些命令、规则、截图可以复用
复盘不是为了显得专业，是为了让下一次少浪费时间。
跑下来以后，我对 Hermes 的判断反而更克制。
Hermes 不是一个装好就能自动赚钱的工具，也不是让人第一天就拥有一家 AI 公司的按钮。它有价值的地方，是把原来散在聊天框里的东西搬到更稳定的位置。
信息筛选进来源台账。偏好和规则进记忆文件。重复流程进 Skill。长任务进 Kanban。不同角色进 Profile。发布结果进复盘。
这些东西单独看都不炸裂。合在一起，它们会把 Agent 从临时帮忙推向长期协作。
我现在更愿意把 Hermes 理解成工作台。
助手解决一句话的问题，工作台解决一周、一月、一个项目的问题。
如果已经跑通 Hermes，我不建议下一步立刻去搭十个 Agent。先做小一点。
让它记住 5 条稳定规则
让它沉淀 1 个重复流程
让它每天筛 5 条有价值的信息
让它把 1 篇文章拆成真实交付状态
让它在卡住时明确 block，不要假装完成
这几件事跑顺了，再谈多 Agent、Gateway、Cron、自动发布，才不会虚。
Hermes 跑通只是第一天。
第七天留下了什么，才决定它有没有真的进入工作。
