第一篇搭了基础框架。现在就是 Codex 干完活了，Claude Code 不知道。
这篇我们就来解决这个问题：怎么让 Claude 自动知道 Codex 已经干完了。

第一章：问 Claude 有什么好办法

有问题，第一反应问 Claude。这也是用学会用 AI 很重要的一个思维：AI First
我有个问题：我在 tmux 里用 Codex CLI 做子 Agent，它在右边 pane 干活。但它干完之后不会通知我，我得自己去看。我想让左边的脚本自动知道 Codex 已经完成了，有什么好办法？
Claude 说这是很常见的 tmux 多 pane 协调问题，给了我从简单到优雅的几种方案：
  • 方案 1：tmux wait-for（推荐）——tmux 内置的信号机制，最干净。右 pane 执行完 Codex 后发信号，左 pane 等待信号，零轮询、零延迟。
  • 方案 2：哨兵文件 + 轮询——右 pane 写个标记文件，左 pane 轮询检查文件是否存在。
  • 方案 3：监控 pane 进程状态——左 pane 获取右 pane 的命令，轮询检查 Codex 进程是否还在跑。
  • 方案 4：tmux capture-pane 检测输出——左 pane 捕获右 pane 屏幕内容，检测 shell prompt 是否回来了。
但是 Claude 最后一句提了一下，不知道我目前已经有实现的方案
因为我的 Claude Code 重新启动了，Claude Code 没有记住。
所以，我下一句，除了询问每个方案有什么坑，还会附带上下文（也就是之前的 SubAgent 实现方式）
Claude 基于我的具体场景，分析了各方案的坑
  • 方案 1（wait-for）：Codex 要是崩了，完成信号发不出来，左边 Claude 就一直傻等，永远卡住。
  • 方案 2（哨兵文件）：同样的问题——Codex 崩了就没人写标记文件。而且脚本里引号套引号，写起来容易出 bug。
  • 方案 3（监控进程）：Codex 可能会启动子进程，进程名变了就认不出来了。
  • 方案 4（capture-pane）：靠检测命令行提示符判断完成，但 Codex 输出里也可能有类似的符号，容易误判。
最好：方案 1 + 方案 3 组合
简单说就是两道保险：Codex 正常干完了，wait-for 立刻通知 Claude；万一 Codex 中途崩了，后台每隔几秒检查一次右边 pane 是不是回到了命令行，发现回来了就补发通知，最多 3 秒就能兜住。
而且再想想，这套机制其实自然而然的就支持多个 Codex 同时干活 —— 每个任务用不同的信号名区分就行。
我继续追问：
基于方案 1 + 方案 3 的组合，Codex 干完活之后，左边的 Claude Code 能够知道吗？具体怎么实现？
收到 Claude 的回复
Claude 的方案：写一个调度脚本，派活、盯着、收结果，一条龙。脚本跑完 = 任务完成。
但关键是 Claude Code 怎么跑这个脚本，这直接决定使用体验：
  • 方案 A：等它跑完。 最稳，但一次只能派一个任务，得排队。
  • 方案 B：丢到后台。 能同时派多个任务，但跑完能不能自动通知回来？我不确定，试试看吧。
我决定：先试试方案 B（后台模式）。如果 Claude Code 能自动收到通知并提示我，那就是最完美的方案。如果不行，再回退到方案 A。
追问 Claude Code
帮我完成方案 B 的改造。
因为我本地有已经完成的 Agent-Team Skill，所以要时不时跟 Claude Code 说一下不要用 Skill
Claude Code 完成改造

实际测试：让 Codex 干个活

改造完成后，我按以下步骤测试后台模式：
派发任务
帮我测试改造后的脚本。用 run_in_background=true 启动一个耗时任务，让 Codex 写一个 Python 脚本下载网页内容并保存，加个 sleep 10 模拟耗时。task-id 用 test-001。
或者用一个更简单口语化的，
跑一个测试看看——用后台模式派个任务给 Codex，看它干完后能不能正常通知回来。
第一次测试翻车了。左边 Claude Code 的 Pane 下面有一个蓝色 Shell，代表了后台正在跑的内容，右边跑完之后，过了一会确实触发了通知，但任务失败了。
为什么翻车？
仔细看发现，第一次发送指令的时候就遇到问题了，导致任务根本没正常启动。这是跑的第二次才成功的，因为我一下子没看到它直接改了临时方案，所以需要找一下问题根源。

第二章：翻车、排查与换方案

测试翻车后，我让 Claude 排查问题。
这里有个关键技巧：
遇到问题，不要让 Claude 直接一股脑去修改。等他排查差不多的时候，记得强制停止它的修改行为。然后问一下：你刚才一通排查，发现了什么问题？如何解决？有什么方案？请跟我描述清楚
这样有几个好处：
  • 避免 Claude 在错误方向上越改越远
  • 让你搞清楚问题本质，而不是蒙在鼓里
  • 节省额度——改之前先确认方案，比来回折腾省多了
Claude 发现的问题
偏技术，看不懂可以跳过，看最终方案，之后做的时候可以直接发最终方案给 Claude
Claude 给的解决方案
原来方案 B（wait-for + 兜底监控）在 Codex 异常退出时还是不够稳。
Claude 建议改用哨兵文件 + 轮询方案
我决定采纳这个方案：
可以，我选择你提供的哨兵文件 + 轮询方案，改完之后，再测试一次。
开始执行，发送完命令之后，Claude Code 这边没有被阻塞，可以正常对话。
右边任务完成之后，正确返回了结果给 Claude Code。

第三章：意外收获——非交互式模式更香

测试过程中，我注意到一个细节：右侧的 Codex CLI 显示跟平时打开的不太一样。
目前的实现，右边是纯 Codex 非交互式模式（直接传命令，不是打开交互界面）。
这其实是个意外收获。作为单 SubAgent 模式的时候，非交互式是最好的：
  • 命令清晰，输入输出明确
  • 不会有交互界面的各种状态问题
  • 方便脚本控制和结果捕获
其实我之前的调用方式用的还是交互式模式，问题比较多，优化了好几次才稳定。写到这里才发现非交互式更稳，文章有意外收获，我也得回去优化之前的版本了。

第四章：遗留问题与总结

遗留问题

右边的 Codex 干完活之后，pane 好像没有自动关闭？可以后续让 Claude Code 优化一下，这里就不写了，你们自己去实践吧。
其实就是跟 Claude Code 说一下问题情况，让它出解决方案，然后开始改，改完让它测试一下，很多事情都可以用这个思路。但是要描述清楚自己看到的问题，以及想要的效果。

这篇做了什么

回顾第二篇的完整历程：
  1. 发现问题：Codex 干完活不会通知 Claude Code
  2. 问 AI 找方案：Claude 给了 4 种方案，推荐 wait-for + 兜底监控的组合
  3. 尝试方案 B：后台模式测试翻车
  4. 排查换方案：强制停止修改，问清问题，改用哨兵文件 + 轮询

  6. 意外收获：非交互式模式比交互式更稳
