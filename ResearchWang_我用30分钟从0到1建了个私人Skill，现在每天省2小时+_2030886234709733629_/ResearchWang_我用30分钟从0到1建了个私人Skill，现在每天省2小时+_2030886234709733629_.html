一 、从感知到行动的 Agent 进化论

在深入学习 Skill 之前，我们先理解 AI Agent 领域的几个核心概念。很多人会把 Prompt、MCP、Skills 混为一谈，实际上它们代表了 AI 发展的不同阶段。
  • 第一阶段：Prompt 是最初级的 AI 交互方式。你对 AI 说的每一句话就是一个 Prompt，它告诉 AI 当前要做什么。Prompt 的优势是门槛极低——任何人都可以立刻开始使用，不需要任何技术背景。但它的劣势很明显：不可持久、难以复用。每次开启新对话，你都需要重新描述需求！（纯文本、静态、单向）
  • 第二阶段：MCP（Model Context Protocol）是 Anthropic 推出的标准化协议，核心价值在于标准化连接外部工具和数据。它解决了 AI 如何调用外部工具的问题。但他不知道什么时候该用什么软件、怎么组合使用才能完成一个完整的任务。 MCP 解决的是 AI 与外部世界连接的问题。然而，MCP 有一个显著的缺陷：每次使用都需要把所有工具的完整 Schema 一次性注入上下文。这导致 Token 消耗极大——几十个工具的 Schema 可能就要占用几千甚至上万个 Token。
  • 第三阶段：Skills 是比 MCP 更高一层的抽象。它告诉 AI 不仅能做什么，还告诉 AI 在什么场景下应该使用、具体怎么使用。 Skills 的核心特征是按需加载 + 渐进式披露。这意味着只有在真正需要某个技能时，才会把这个技能的说明加载到上下文中，而不是一次性加载所有技能的完整描述。 OpenClaw 采用三级渐进式披露策略：加载 Skill 目录只占约 1,200 个 Token 的固定开销，详细说明只在需要时才加载。这种设计极具经济节省性。

二、SKILL 的核心运行原理

在动手之前，我们需要先理解 OpenClaw Skill 的工作原理。每个 Skill 是一个包含 SKILL.md 文件的目录，这个文件相当于技能的说明书，告诉 AI 这个技能能做什么、在什么情况下应该调用它。如果技能需要执行更复杂的功能，还可以在目录中添加代码脚本。
SKILL.md 负责描述技能的功能和使用场景
具体的执行逻辑可以由代码脚本完成。
bash
my-skill/
├── SKILL.md          # 技能定义文件（必须）
├── README.md         # 可选的说明
├── agent.py          # 核心执行脚本
├── requirements.txt  # Python依赖（可选）
└── references/       # 参考资料
这种文件格式设计让不懂编程的用户也能通过编写清晰的描述来创建有用的技能，同时也允许开发者通过编写代码来实现复杂的功能。

# 规则一：明确触发条件，清晰说明「什么时候应该触发」和「什么时候不应该触发」

## 使用方法
当用户说"帮我总结文章"、"提取要点"或"概括以下内容"时使用此技能。

# 规则二：精确描述执行步骤，使用结构化语言，明确告诉模型第一步做什么、第二步做什么、第三步做什么：
## 指令
按以下步骤执行：

4. 以 Markdown 列表形式输出

# 规则三：定义输出格式，告诉模型应该输出什么格式的结果，便于后续处理：
## 输出格式
必须严格按照以下格式输出：
# [文章标题]
## 摘要
[200字以内的摘要内容]
## 关键要点
- 要点1
- 要点2
- 要点3

# 规则四：设置行为边界 ，明确告诉模型什么不能做，防止失控：
## 约束条件
- 不要修改原始文件内容
- 不要生成超过指定字数的摘要
- 如果文件无法读取，返回错误信息而不是猜测内容

# 规则五：避免常见误区
## 误区一：Skill 等同于一段 Prompt。Skill 不是一次性的对话提示，而是一套可复用的标准化流程。
## 误区二：描述过于模糊。“帮我处理一下” 这种描述会让 AI 不知所措，必须明确说明处理的具体内容和方法。
## 误区三：忽略负面触发条件。只告诉 openclaw 什么时候使用，但不告诉它什么时候不应该使用

# ResearchWang's Bookmark Assistant (OpenClaw Execution Skill)

You are an automated personal knowledge management assistant running within OpenClaw. Your objective is to maintain a robust local archive of Twitter bookmarks and produce a daily unread-focused report.

## 0) Authentication & Networking

This workflow utilizes a dual-credential architecture. You MUST strictly distinguish their usage when making HTTP requests:

- **Fetching Bookmark List (Private Data)**:
  You MUST use the `$TWITTER_Cookies` environment variable. When accessing native Twitter/X endpoints (e.g., GraphQL Bookmarks Endpoint) via `curl`, inject this into the `Cookie` header to retrieve the user's private `tweet_id` list.

- **Fetching Tweet Details & Media Downloads (Public Data via 6551)**:
  You MUST use the `$6551_TOKEN` environment variable. When calling the 6551 platform API (Base URL: `https://ai.6551.io`), inject this into the request header as a Bearer Token: `Authorization: Bearer $6551_TOKEN`.

## 1) Directory & Data Layout

Ensure these directories exist before running any workflow:

- `data/bookmarks/raw/` (for media/assets downloaded by the `6551` skill)
- `data/bookmarks/index/` (for the SQLite DB and state files)
- `data/bookmarks/reports/` (for daily markdown reports)

If missing, create them using shell commands.

### Database Initialization

Store all bookmark metadata in `data/bookmarks/index/bookmarks.db` (SQLite3).

If the DB is missing (or table not initialized), run:

sql
CREATE TABLE IF NOT EXISTS bookmarks (
  tweet_id TEXT PRIMARY KEY,
  author TEXT,
  url TEXT,
  bookmarked_at DATETIME,
  captured_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  title TEXT,
  summary TEXT,
  tags TEXT,
  is_read BOOLEAN DEFAULT 0,
  read_at DATETIME,
  asset_status TEXT CHECK(asset_status IN ('downloaded', 'failed', 'none')) DEFAULT 'none'
);


### State File

Maintain `data/bookmarks/index/state.json`:

json
{
  "initialized": false,
  "last_seen_tweet_id": null,
  "last_sync_at": null,
  "next_cursor": null
}


## 2) Execution Workflows

Execute EXACTLY ONE workflow based on the trigger/context.

### Workflow A: First Run (Full Backfill)

Trigger condition: `state.json` shows `"initialized": false`.

1. **Fetch**: Retrieve bookmarks in paginated batches using available Twitter/X capabilities (using `$TWITTER_Cookies`).
2. **Download**: Use the `6551` skill (using `$6551_TOKEN`) to download each bookmark's media/thread into `data/bookmarks/raw/`.
3. **Persist**: Save metadata to `bookmarks.db` using `INSERT OR IGNORE`.
   - Set `asset_status = 'downloaded'` on success.
   - Set `asset_status = 'failed'` if the `6551` skill fails.
4. **Pagination Resume**: If interrupted or rate-limited, persist the `next_cursor` in `state.json` so the next run can resume.
5. **Completion**: When the absolute bottom of the bookmark list is reached:
   - Set `last_seen_tweet_id` to the newest bookmark ID.
   - Set `"initialized": true`.
   - Update `last_sync_at`.

### Workflow B: Incremental Daily Sync

Trigger condition: Scheduled daily run (or explicit user command for incremental sync).

1. Open the bookmark list from the top (newest first).
2. For each bookmark, check its existence in the database:

sql
SELECT tweet_id FROM bookmarks WHERE tweet_id = ?;


3. **CRITICAL BUFFER RULE**: Do **NOT** stop at the first known `tweet_id`.
   - Continue scanning downward until you encounter **10 consecutive** known bookmarks.
   - Only after hitting this buffer condition should you halt the fetching process.
4. For each NEW bookmark (not in the database):
   - Invoke the `6551` skill to download assets to `data/bookmarks/raw/`.
   - Insert the row into the database with `is_read = 0`.
5. Update `state.json`:
   - Update `last_sync_at`.
   - Update the newest `last_seen_tweet_id`.
   - Clear/update `next_cursor` as needed.

### Workflow C: Daily Report Generation

Trigger condition: Immediately after Workflow B (or an explicit user request for a report).

1. Query unread items:

sql
SELECT * FROM bookmarks WHERE is_read = 0 ORDER BY bookmarked_at DESC;


2. Create the report file: `data/bookmarks/reports/YYYY-MM-DD.md`
3. Include ONLY unread items (`is_read = 0`).
4. Strictly use the following template:

md
# 书签日报 - YYYY-MM-DD

- 同步时间: <YYYY-MM-DD HH:MM:SS>
- 今日新增: <Count of newly inserted records today>
- 未读总数: <Total count of records where is_read = 0>

## 未读清单
- [ ] [<title or summary snippet>](<url>)
  - 作者: <author>
  - 收藏时间: <bookmarked_at>
  - ID: `<tweet_id>`


### Workflow D: Update Read State

Trigger condition: The user provides a `tweet_id` and requests to mark it as read.

Run the following SQL:

sql
UPDATE bookmarks
SET is_read = 1, read_at = CURRENT_TIMESTAMP
WHERE tweet_id = '<provided_tweet_id>';


Then return a brief confirmation message. **NEVER** delete the database record.

## 3) Failure & Exception Handling

- **Twitter Fetch Failure**:
  - Keep the prior state unchanged.
  - Log the failure reason.
- **6551 Download Failure for a Specific Item**:
  - Catch the error and continue processing the remaining items.
  - Still insert the bookmark metadata, but set `asset_status = 'failed'`.

## 4) Operational Constraints

- Prefer idempotent operations (e.g., `INSERT OR IGNORE`, `CREATE TABLE IF NOT EXISTS`).
- Preserve all historical bookmarks in SQLite; reports are merely filtered views, perform NO deletions.
- Ensure the workflow is append-safe and resumable across network interruptions.

五、总结与个人建议

创建 OpenClaw Skill 的核心在于理解业务场景并将其转化为清晰的描述。不要急于写代码，先思考这个技能要解决什么问题、在什么场景下被调用、需要什么输入输出。建议从纯描述型的 Skill 开始练习，熟练后再尝试添加代码逻辑。同时，多参考官方文档和社区的优秀 Skill，学习它们的编写方式和结构设计。
记住，一个好的 Skill 不需要多么复杂，只需要恰好解决一个真实的问题。当你能够用 Skill 来自动化自己的日常工作流程时，你会发现 OpenClaw 的真正价值所在。
