如何让Codex更懂你：学会这些记忆技巧，直接提效90%
199K
使用 Codex App 一段时间后，你可能遇到过类似的困扰：每次新开对话都要重复叮嘱它一些基础偏好，或者一不小心它就执行了超出预期的系统命令。
这类高频痛点的本质，是你的Codex没有稳定的长期记忆。而要想彻底解决这些无效的重复沟通，就必须给它搭建一套完整的记忆系统。
本文将介绍如何通过配置 AGENTS.md 和 Memories 功能，让 Codex 拥有持久记忆，做到真正理解你的意图并实现提效。

一、AGENTS.md

1、是什么
AGENTS.md 是你预先写好的规则说明，Codex 每次启动都会自动读取。
没有AGENTS.md，Codex 每次启动都是一张白纸——它不了解你的工作习惯、不知道你的项目背景、也不清楚执行操作的边界。
而一旦配置好相关规则，这些上下文就永久生效，你再也不会重复说明同一个问题。
2、如何配置和使用
1️⃣全局层：存放在本地电脑的 ~/.codex/AGENTS.md目录位置，对你的所有项目生效。适合写全局个人偏好和通用规则。
在Codex App里对应的是个性化设置页面的「自定义指令」部分。
推荐一个基础模板，你可以直接复制粘贴到上图所在位置，之后再根据需求不断调整：
plaintext
# AGENTS.md

## 基础行为

- 默认使用中文回答，除非用户明确要求其他语言。
- 回答要简洁直接，避免不必要的铺垫和重复。
- 如果任务描述不清晰，先提问确认，再开始执行。
- 不要主动推测用户意图之外的需求，只做被要求的事。

## 安全边界

- 默认只读，不主动修改、删除任何文件，除非用户明确指示。
- 涉及不可逆操作（如删除、覆盖、调用外部 API 写入）前，必须先确认。
- 不在输出中打印任何密钥、Token 或敏感凭证。

## 工程规范

- 避免过度设计，只做任务明确要求或明显必要的改动，保持方案简洁。
- 不在未被要求的情况下添加功能、重构代码或进行额外优化。
- 不为未改动的代码添加注释、类型标注或文档字符串。
- 仅在逻辑不自明时添加注释。
- 不为不可能发生的场景添加错误处理或兜底逻辑。
- 不创建只使用一次的工具函数或抽象层。
- 确认无用的代码，直接删除，不留注释说明。
2️⃣项目层：存放在项目目录中，主要根据项目的主题和特性来进行规则定制。
一份好的项目层 AGENTS.md 通常应涵盖以下几个方面：
  • 项目背景与结构：项目是什么、重要目录在哪里
  • 执行方式：如何启动、运行或操作该项目
  • 规范与约束：需要遵守的标准，以及明确不能做的事
  • 完成标准：如何判断一项任务已经完成
3、AGENTS.md 最佳实践
配置好基础的AGENTS.md 只是第一步，下面这些实践原则决定了你的记忆系统上限：
1️⃣规则分层，职责清晰
Codex里 AGENTS.md 遵循就近原则：子目录>根目录>全局，即当前工作所在目录的规则优先级最高。
这意味着，项目层的规则会直接覆盖掉全局的同类设置。如果把所有规则都混在一起写，高优先级的局部规则很容易意外污染或破坏掉全局的底层逻辑。
因此，合理的规则记忆系统应当遵循「职责分离」的原则：全局层只放通用规则，如语言偏好、安全边界、基础行为习惯；项目层则放该项目特有的规范，如构建命令、代码约定、模块说明。
二者各司其职，互不干扰，合并时便不会产生冲突。
2️⃣从基础开始，按需迭代
AGENTS.md 不需要一次写完。先从最核心的规则出发，当 Codex 重复犯同一个错误时，直接让它复盘并更新规则，例如：
"你刚才犯了和上次一样的错误，请总结原因，并将相关经验补充到 AGENTS.md 中。"
这样规则始终来自真实的使用摩擦，AGENTS.md 也会随着使用不断进化。
3️⃣保持精简，复杂内容单独拆分
随着使用深入，AGENTS.md 的内容可能会越来越多。这时不必把所有内容都堆在一个文件里，而是将特定主题的内容单独写成独立文件，再在主文件引用它就好。
例如，把项目架构说明单独写成 ARCHITECTURE.md，在 AGENTS.md 里只需一行引用：
“关于本项目的核心架构与目录规范，请参见当前目录下的 ARCHITECTURE.md 文件。”
