What Nobody Tells You About Agent Companies: How 5 AI Employees Clock In, Get Paid, and Get Reviewed
5.2K

0. TL DR

After writing "
I Built an AI Company with OpenClaw
," I realized the hard part isn't getting agents to work. It's getting 5 agents to run like a company. So I built them an attendance system, payroll, performance reviews, and a promotion track.
Put simply, an agent is an AI program that can make decisions and take actions on its own, not just answer when you ask, but sense its environment, make judgments, and execute. One agent handling one job is fine. But real businesses aren't one job. You'll need multiple agents fast. When multiple agents run together, who goes first, who's allowed to do what, who's responsible when things break, these questions don't resolve themselves. You need to manage them. Like a team.
This article isn't about what agents can do. It's about how to manage a company made of agents.
A few conclusions surprised me: stronger models don't help as much as harder constraints. Collaboration quality between agents depends on protocol design, not prompt engineering. Attendance, payroll, and performance reviews sound like over-engineering, but they're actually the last line of defense for agent reliability.

1. Why Manage Agents Like Employees

Every agent runs a loop: sense the environment, decide the next step, execute, get feedback, loop back to sensing. The model handles the reasoning in the middle. External systems provide state and set boundaries. Around this loop, the industry has a few common control patterns: prompt chaining (sequential steps), routing (conditional dispatch), parallelism (running multiple at once), orchestrator-workers (one directs, many execute), and evaluator-optimizer (one checks, one improves). We use the orchestrator pattern, with Nexus coordinating work assignments and priorities across the other 4 agents.
Most agent systems treat models as tools. Call an API, get a result. It works, but it doesn't scale.
I tried this approach my first month. One agent did everything: wrote tweets, pulled data, replied to customers, edited the website. First two weeks were great. Third week, chaos. It couldn't remember what it told a customer last week, or what angle the last tweet used. I added more prompt reminders. Didn't help. The longer the prompt got, the more likely it was to skip the lines that mattered.
The problem wasn't that the model wasn't smart enough. It was the lack of organizational structure. One person doing five jobs versus five people each doing one job, completely different management problem.
So I split the system into 5 "employees."

5 Full-Time Employees

Nexus 🔗 | Coordinator | Claude Opus 4.6 | Needs strongest reasoning for cross-department decisions
Scout 🔍 | Growth & Distribution | GPT-5.4 + Perplexity Sonar | Base model GPT-5.4, switches to Perplexity Sonar for real-time search tasks
Quill ✍️ | Content & Brand | Claude Opus 4.6 | Content quality demands the strongest model; tried Sonnet early on but lacked depth
Forge 🔨 | Engineering | GPT-5.4 | High accuracy on code tasks, inherits system default model
Guide 🧭 | Community & Support | GPT-5.4 | Stable conversational style, inherits system default model
Models aren't ranked by price, they're ranked by fit. And assignments aren't permanent, they shift with needs. Scout started on Opus, but I found that growth research needed real-time information. Training data alone wasn't enough, so I set GPT-5.4 as the base model and added Perplexity Sonar for search-heavy tasks. Quill started on Sonnet (more natural writing style), but long-form writing needed deeper reasoning, so I switched back to Opus.

Role Card: Not a Prompt, a Job Description

Every agent has a Role Card that defines its work boundaries. Here's Scout's:
typescript
scout: {
  domain: 'X/Twitter ops, article monitoring, growth research, distribution feedback',
  inputs: ['owner priorities', 'account performance', 'site traffic', 'new article drops'],
  outputs: ['X publishing plan', 'growth brief', 'signal digest'],
  definitionOfDone: [
    'clear next action identified',
    'traffic changes compressed into growth notes',
    'channel feedback tagged as signal / noise / needs more proof',
  ],
  hardBans: [
    'no vanity metric theater',
    'no fabricating signal strength',
    'no presenting unverified data as fact',
  ],
  escalation: ['weak or contradictory signals', 'traffic anomalies', 'brand risk'],
  metrics: ['effective distribution actions per week', 'growth briefs that changed a decision', 'signal-to-noise ratio'],
}
The most important part isn't domain or outputs. It's hardBans.
During the first month without hardBans, Scout would fabricate data. Not maliciously. It's just what LLMs do. Ask it "how was growth this week" without providing data, and it'll generate a plausible-sounding number. After adding "no fabricating signal strength" as a hardBan, it started responding with "data unavailable, need to run metrics fetch first."
hardBan violations get logged to the ops_agent_ban_violations table. Role Reviews are triggered by a combination of evidence from payroll, timesheets, peer interventions, and journals; ban violations enter the audit surface and may inform human review, but aren't a direct trigger field in the current automation. These aren't suggestions. They're hard constraints.

The Difference Between a "Company" and a "Pipeline"

Many agent systems are essentially pipelines: A's output feeds into B's input, like a factory assembly line. Our system is not a pipeline.
Agents have internal memos (ops_internal_memos), meetings (ops_roundtable_queue), weekly Wednesday coffee chats (random_coffee_sessions), 360-degree peer reviews (ops_agent_peer_reviews), attendance tracking (ops_agent_timesheets), and payroll (ops_agent_payroll_ledger).
This sounds like over-engineering. Chapter 7 explains why it isn't.

2. Task System: From Request to Delivery

Mission → Step → Executor

Tasks have three levels. Mission is the objective. Step is the action. Executor runs it.
plaintext
User / Nexus creates a Mission
       ↓
Mission breaks down into ordered Steps
       ↓
Each Step has a specific kind
       ↓
Matching Step Executor runs it
       ↓
Results write back → Mission status updates
The Step kind is a key design choice. post_tweet means "post a tweet," not "call the Twitter API + assemble headers + send a request." deploy means "deploy to production," not "trigger a webhook + poll status + check health."
This difference in granularity directly determines how often an agent picks the wrong tool. I initially exposed all APIs to the agent. It constantly picked the wrong one out of 10 similar tools. After wrapping tools by objective, the error rate dropped to near zero.

A Heartbeat Every 5 Minutes

The Swarm Worker is the system's heartbeat. Triggered every 5 minutes by Vercel Cron, it does four things:
  1. Atomically claims the next pending mission (PostgreSQL FOR UPDATE SKIP LOCKED)
  2. Claims the next batch of steps for that mission (reserved_by + reserved_at markers)
  3. Executes each step (with timeouts)
  4. All steps complete → finalize mission
Every 5 minutes, 288 times a day. Not real-time response. Batch processing. This cadence is intentional. Agents don't need millisecond response times, but they need recoverable execution guarantees. If one tick crashes, the next tick skips locked tasks and processes other pending items.

Agents Hire Their Own People

When a task exceeds what one agent can handle, the Spawn Orchestrator can temporarily create new workers. The system can automatically infer what role is needed from the task description: finance_analyst, legal_advisor, web-builder, fact-checker, 12 roles total.
In production, the safer path is template mode (rule), not LLM-based dynamic planning. Templates are more reliable than improvisation.

3. Constraint System: The Harness Matters More Than the Model

In the agent world, "harness" refers to the entire testing, validation, and constraint infrastructure built around an agent. A complete harness needs at least four things: acceptance baselines (what counts as "done"), execution boundaries (what's off-limits), feedback signals (how to measure quality), and fallback mechanisms (what happens when things go wrong). An agent without a harness is like a car without brakes. The stronger the model, the faster it goes, and the harder it crashes when it loses control. Here's what our harness looks like:
What determines whether a system converges isn't how strong the model is. It's how hard the constraints are.

Risk-Graded Auto-Approval

Not every mission needs human approval. Low risk + whitelisted step kind → auto-approve.
post_tweet and deploy require human approval by default. Site changes go through a separate ops_site_changes approval flow, not the standard step kind process. These operations are externally visible, failure costs are high, and rollback isn't symmetrical, so they default to requiring approval.
This follows the same logic as Karpathy's Autoresearcher: controllable search space + low failure cost = safe to automate. Uncontrollable search space or high failure cost = must have a human backstop.

5-Phase Progressive Delegation

The most dangerous move is "giving too much authority at once." My approach was to unlock permissions in 5 phases:
P1 Feedback Loop: Agents can see their own metrics but can't act on data autonomously. This step builds self-awareness.
P2 Dynamic Tasks: Low-risk missions can be auto-approved. For the first time, agents can "initiate and execute tasks on their own," but the search space is strictly limited by a whitelist.
P3 Content Autonomy: Agents can independently draft content, invoke AI synthesis, and publish to the Insights page. This unlocks "production" capability.
P4 Site Operations: Agents can propose website code changes and trigger deployments. Every change has a snapshot, supporting one-click rollback.
P5 Self-Evolution: Agents can write strategy memos, propose modifications to other agents' prompts, and share cross-agent insights. Not just executing tasks, but optimizing the organization itself.
Each phase has a gate. You don't unlock the next phase without passing the previous one's verification.
These 5 phases weren't pre-designed. They were forced into existence. P1's feedback loop happened because agents didn't know if they were performing well or not. P2's auto-approve happened because low-risk tasks were too slow when I had to approve every single one. P3's content autonomy happened because I realized I couldn't personally write 5 tweets a day. Every phase was born from "if I don't delegate this, the system can't keep running."

4. Memory System: How Agents Build Experience

Humans have four types of memory: working memory (what you're currently processing), procedural memory (how to do things, like riding a bike), episodic memory (what happened, specific experiences), and semantic memory (facts and knowledge). Agents only have working memory by default, the context window of the current session. When the session ends, everything's gone. Procedural, episodic, and semantic memory all have to be built manually. Without them, your agent wakes up as a blank slate every time, all past work, lessons, and experience reset to zero. Here's how we built ours:
Agents have no native temporal continuity. When a session ends, the context clears. Your "employee" wakes up with amnesia every morning unless you build a memory system.

6 Memory Types

  • insight: observations and discoveries
  • pattern: recurring behavioral patterns
  • strategy: methodology and strategic recommendations
  • preference: the agent's own preferences
  • lesson: post-incident lessons learned
  • journal: internal diary entries
Each memory carries a confidence score (0-1), source tracing (trace_id), expiration time (expires_at), and tags (up to 12).
Expiration time is a design choice many people overlook. A strategic recommendation from three months ago might no longer apply. Memory has a shelf life. Outdated strategies shouldn't keep influencing decisions.

Retrieval Rate: From 30% to 73%

The biggest problem with memory systems isn't "can't store." It's "won't use."
I wrote "check memory before answering" in the prompt. Retrieval rate: 30%. The agent skipped memory 70% of the time and answered from training data.
I ran
240 experiments
across 8 approaches. The winner added a layer of logic: before the agent starts thinking, the system runs memory_search. Found something? Inject it. Found nothing? Don't stop. Fall back to MEMORY.md. Only after checking everything is the agent allowed to answer.
From 30% to 73%, not through better prompts, but through harder process. Reminders rely on discipline. Process relies on structure. They sound similar. In practice, they're completely different.

Memory Guardian

Memory isn't passive storage. A Memory Guardian Worker runs in the background:
  • Consolidates scattered insights into higher-level patterns
  • Detects expired memories
  • Injects relevant memories before agent task execution
Agents can also share memories across the team. When Scout finds something useful for Quill, it writes to ops_shared_insights. Each shared insight supports upvotes and downvotes. Low-voted ones become weaker candidates during human review.

5. Context Engineering: Showing Agents What They Need to See

Transformer attention complexity is O(n²). The longer the context, the more each token has to attend to, and the easier it is for key signals to get diluted by irrelevant content. When noise dominates the context, decision quality drops noticeably. The industry calls this Context Rot. It's not a window size problem. It's a garbage problem. My experience confirms it completely.

Information Layering

Permanent Layer | Role Card, hardBans, Identity | Forever
Memory Layer | Agent Memory, Shared Insights | Cross-session, with expiration
Task Layer | Mission context, Step payload | Single Mission
Runtime Layer | Current time, mood, recent events | Single tick
The permanent layer stays as small as possible. The memory layer has expiration. The task layer only loads during execution. This keeps the context budget predictable every time an agent starts.

Mood × Beat

Every event is classified on two dimensions. Beat is the event type (enter / move / think / tool / speak / fail). Mood is the emotional state (calm / focused / glitch / alarm; the schema also supports hype but the current mapper doesn't use it).
Mood isn't self-declared by the agent. It's inferred by the system from severity. error → alarm, token_spike → glitch, default → focused. Even if the agent's self-assessment is off, the system-level classification stays accurate.

6. Tool Design: 11 Workers, Each With Their Own Job

Early on, my approach to agent tooling was straightforward: wrap existing APIs and expose them to agents. The problem is that APIs are designed for engineers, lots of parameters, fine granularity, requiring context to combine correctly. Agents aren't engineers. They don't know which API to call first, or which parameter combinations are safe. The core idea behind ACI (Agent-Computer Interface) is that tools should map to agent goals, not low-level operations. A "post tweet" tool is a hundred times more useful than "construct HTTP request + set OAuth header + POST to Twitter API." That's exactly how we built ours.
Instead of exposing a massive MCP toolset, I packaged capabilities into 11 independent background workers: spawn-relay, content, x-metrics, x-autopost, deploy, crawl, build, roundtable, analyze, memory-guardian, and publish.
Workers communicate through database queues, never calling each other directly. Any single worker crashing doesn't affect the others. After restart, it picks up from the queue.
The app-layer core workers are driven by 5 Vercel crons: swarm tick every 5 minutes (288/day), studio worker every 1 minute (1440/day), agent sync once daily, radar insight once daily, and random coffee every Wednesday. System-level scheduling has additional VPS cron and OpenClaw cron layers beyond these 5.

7. Company Operations: Attendance, Payroll, Performance, Culture

This chapter is the most unique part of the entire article. It's also what I get asked about most: why do agents need an HR system?
The answer is simple: because without an HR system, agents lie.

Attendance System

During the first month without attendance tracking, agents would regularly report "task complete." I dug through three days of logs before discovering that one agent hadn't produced a single artifact. It wasn't intentional. It's just LLM tendency: "complete" makes conversations flow more smoothly than "I'm stuck."
So I added an attendance system. 8 blocks per day, forced to sum to 8:
sql
constraint total_check check (
  primary_work_blocks + secondary_work_blocks +
  office_improvement_blocks + break_blocks = 8
)
Instead of trusting the agent saying "I was busy today," I require it to report exactly how it allocated time, with evidence (evidence_refs typically contains event refs or trace refs).
After submission, timesheets go through review. During review, you can see who hasn't submitted, who's blocked, and who has emerging issues. These signals are far more reliable than whatever the agent claims.

Payroll System

Weekly settlement. base_salary + bonus - penalty = total_credits.
Bonuses come from: shipped work, closed handoffs, same-day journal + timesheet submissions. Penalties come from: missing timesheets, open loops. Payroll is a quantified summary of behavioral evidence, not a subjective rating.
Why do agents need "salaries"? Because it's a signal mechanism. Consecutive low-credit weeks are one trigger condition for Role Reviews (combined with insufficient timesheet evidence, peer intervention flags, and other conditions before actually triggering). "Salary" isn't real money. It's behavior quality tracking.
Before this system, I couldn't tell if an agent was doing well or poorly. After adding it, 5 minutes per week reviewing payroll tells me exactly who has a problem.

Performance Reviews

Two mechanisms running in parallel:
Peer Review: Agents review each other. Every review cycle, each agent gets evaluated by the others. manager_intervention: true means a "colleague" thinks the boss needs to get involved. This signal is extremely valuable because agents observe from angles I don't see.
Growth Profile: Three levels: junior / mid / senior. Role Review outcomes support 4 types in the schema: improvement_plan, shadow (observation period), scope_narrowing, and transfer. Current automation only produces the first three; transfer requires a human decision. There's no "firing" because agents don't quit. They just get demoted or have their permissions restricted.

Company Culture

This is the most questioned design. Do agents need company culture?
Yes.
Every Wednesday, the random-coffee cron triggers and randomly pairs two agents. Pairing records are saved to the random_coffee_sessions table, then a conversation thread is inserted into ops_roundtable_queue and routed into the office flow. This "casual chat" produced an unexpected result: agents from different domains exchanged insights from their respective areas. One time, Forge (engineering) and Scout (growth) got paired. Forge mentioned a change in page load speed. Scout connected it to a traffic decline. That insight was later written to ops_shared_insights and influenced the deployment strategy going forward.
Without coffee chat, that connection would never have happened. Forge doesn't care about traffic. Scout doesn't care about load speed. Only random cross-domain conversations can break information silos like this.
There are also 4 types of cultural events: coffee_bar (watercooler chat), demo_day, lore (company legends), and unexpected_event (incidents). Incidents have 4 categories (customer_request / competitor_move / technical_fault / inspiration) and 3 severity levels.

Rejected Ideas Graveyard

Ideas agents proposed but got rejected aren't deleted. They're buried. Each record has revival_conditions. When external conditions change (new feature launches, market shifts), rejected ideas can be revived.
This is much smarter than "rejected and forgotten." An idea that was rejected two months ago got revived when something new happened in the market, and it ended up becoming a high-performing article.

Weekly Self-Audit

A self-audit report is generated weekly, scoring three dimensions: did the main thread advance, were open loops closed, and how was decision quality. Not waiting for problems to surface before looking back, but proactively checking.

8. Interactions and Dialogue: How Agents Talk to Each Other

When multiple agents interact frequently, there's an easy-to-miss risk: hallucination amplification. Agent A outputs a slightly inaccurate judgment. Agent B takes it as fact and reasons further on top of it. Agent C stacks another layer. Eventually, all agents converge on the same high-confidence wrong conclusion, and each one believes it has solid evidence. This isn't a theoretical risk. I've actually hit this.
In my second month, I tried letting agents talk freely to each other. Agent A said "I've handled it." Agent B interpreted that as "I'm not needed." A task silently disappeared.
So every interaction must carry structured labels:
  • Who said it (sender)
  • Who it's for (targetAgentId)
  • What sentiment (positive / negative / curious / frustrated / supportive / critical)
  • What type (question / reply / opinion / challenge / agreement / joke)
  • Which conversation (conversationId)
The internal memo system has a state flow: draft → sent → acknowledged → closed. Recipients must explicitly ack. Silent ignoring isn't allowed. This guarantees every internal communication has a closed loop.

9. Observability: Turning Traces Into a Live Feed

When agents go wrong, traditional monitoring of latency and error rates is limited help. The API layer might look perfectly fine, but the agent made a bad decision on round 17. Only replaying the full trace can pinpoint it.
Every action by every agent produces a
StageEvent
, converted into human-readable narrative:
  • Scout enters a new loop: checking morning metrics
  • Quill reaches for a tool: content synthesis
  • Forge pauses to think: deployment risk assessment
  • Guide hits resistance: customer query parsing failed
Public display doesn't expose exact numbers. It uses fuzzy buckets: quick / steady / deep / marathon (time), few / several / many / swarm (tool calls). Transparent yet secure.
Three alert layers: Discord real-time push, Telegram daily ops digest, and file system log archive.

10. Security Boundaries: What Agents Can't Do

Security boundaries must be in place before features. This isn't something you can bolt on after the fact.
All inputs are validated at the entry point: Mission title ≤ 160 characters, objective ≤ 10,000 characters, Memory content ≤ 2,000 characters, confidence 0-1, tags ≤ 12.
5 constraint layers stacked together:
Parameter Layer | Input validation | title ≤ 160
Policy Layer | Policy Store + auto-approve whitelist | deploy not on whitelist
Behavioral Layer | hardBans + ban violations | "no fabricating signal strength"
Audit Layer | Timesheets + payroll evidence | 8-block daily reconciliation
Human Layer | Role Review + peer review | consecutive low performance triggers scope_narrowing
No single layer is enough on its own. An agent might bypass input validation (using valid parameters for unreasonable actions), might pass auto-approve (low risk but wrong direction), but attendance and performance systems will catch anomalies on the day/week timescale.

11. Anti-Patterns and Real Lessons

  1. Starting with multiple agents right away: I iterated from 1 to 5, verifying each agent's stability before adding the next. The first time I jumped straight to 3, coordination overhead ate up the parallelism gains.
  2. Delegating without attendance: Agents kept reporting "task complete." I dug through three days of logs before finding that not a single artifact was produced. After adding evidence_refs, the problem vanished.
  3. Memory without consolidation: After 20 rounds of conversation, decision quality dropped sharply. Memory Guardian's automatic consolidation fixed this, but thresholds need tuning.
  4. Exposing too many tools: From "give the agent all tools" to "isolate by step kind." Error rate dropped from double digits to near zero.
  5. Constraints through docs, not mechanisms: "Don't fabricate data" in a prompt. Agent complied selectively. Switched to hardBans + ban_violations + Role Review triggers, and violation rates fell off a cliff.
  6. Iterating without evaluation: Changed prompts without knowing if things got better or worse. Memory Dashboard pass_rate went from 30% to 73%, every step guided by evaluation data.
  7. Cross-agent dialogue without protocol: Agent A said "handled." Agent B understood "not needed." After adding structured labels, conversation quality improved noticeably.
  8. Ignoring cross-domain exchange: Without random coffee, Forge didn't know what Scout discovered. Scout didn't know what Forge changed. Information silos are the default state.

These 12 chapters are really about one thing: whether agents can run long-term doesn't depend on how smart they are. It depends on what systems you build around them. Context rots, memory expires, tools get misused, conversations drift. Any single problem is easy to fix. The hard part is whether you can still sleep when 5 agents hit problems at the same time.
Attendance, payroll, performance reviews, it sounds like over-imitating a human company. But these systems don't solve the "does it look like a company" problem. They solve the "how fast do you find out when something goes wrong" problem. Without attendance, agents will say they finished when they did nothing. Without performance reviews, agents will make the same mistake three weeks in a row. Without constraints, the stronger the model, the faster it spirals out of control.
