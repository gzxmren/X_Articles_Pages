2026年最值得一试的5个Claude Code 插件，看看能否解决你的痛点
1.2K

Claude Code 插件不是越多越好。

真正值得装的插件，应该能补一个明确的工作流短板。
下面这 5 个工具，各补一个口子。别一起装，先挑最疼的那个。
Claude Code 插件不要全装，先按痛点选一个

1. Impeccable：前端能跑之后，再让它查一遍设计

你让 Claude Code 写前端，功能通常很快能跑。但页面经常一眼像 demo：间距松、层级乱、按钮没状态、卡片到处套卡片。
Impeccable 解决的是这个问题。它不是设计师，也不是 UI 模板库，而是把“页面不好看”拆成 Claude Code 能继续执行的设计任务，比如视觉层级、对比度、间距、响应式和交互细节。
入口有两个：
  • 官网：
    https://impeccable.style
  • GitHub：
    https://github.com/pbakaus/impeccable
推荐安装方式是在项目根目录跑：
bash
npx impeccable skills install
这条命令会识别你用的是 Claude Code、Cursor、Gemini CLI、Codex CLI 等工具，然后把对应 skill 装到合适的位置。
Claude Code 用户也可以走插件市场：
text
/plugin marketplace add pbakaus/impeccable
装完之后，别一上来就让它重做整个网站。先查一个小区域：
text
/impeccable audit the header
或者查一个完整页面：
text
/impeccable audit the homepage
正常结果是一份设计问题清单，里面会指出视觉层级、间距、对比、响应式、按钮状态这些具体问题。接下来再让 Claude Code 按建议改代码。
Impeccable 把粗糙页面变成设计修复清单
拿一个已有前端项目试最省事。打开项目后先装 skill，再在 Claude Code 里只查一个区域：
bash
npx impeccable skills install
text
/impeccable audit the pricing section and list the top 5 visual fixes
没有现成项目，也可以临时做一个很粗糙的 HTML 页面，让它有东西可查：
bash
mkdir impeccable-demo
cd impeccable-demo
cat > index.html <<'HTML'
<section style="padding:80px">
  <h1 style="font-size:48px">AI Notes</h1>
  <p style="color:#999">All your notes in one place.</p>
  <button style="padding:8px 12px">Start</button>
</section>
HTML
npx impeccable skills install
然后在 Claude Code 里跑：
text
/impeccable audit index.html
这一轮跑完，应该能看到设计问题清单，或者得到一组可继续交给 Claude Code 修改的建议。它的价值不在安装过程，而在它能把“页面不好看”拆成可执行的修改项。
Impeccable 更适合放在“功能写完之后，发布之前”。页面还没跑起来时别急着用它；没有真实页面，它只能空谈设计方向，价值会打折。

2. Caveman：Claude Code 太能说时，给它降噪

Claude Code 用久了，真正累人的不一定是等它写代码，而是读它的长回答。
Caveman 做的事很窄：让 Claude Code 输出更短。少客套，少铺垫，少重复，直接给结论、改动和风险。
入口在这里：
  • GitHub：
    https://github.com/JuliusBrussee/caveman
  • 安装说明：
    https://github.com/JuliusBrussee/caveman/blob/main/INSTALL.md
Claude Code 一行安装：
bash
claude plugin marketplace add JuliusBrussee/caveman && claude plugin install caveman@caveman
如果你也用 Codex CLI，可以单独装到 Codex：
bash
npx skills add JuliusBrussee/caveman -a codex
Claude Code 里装完一般会自动生效。你可以直接给它一个日常任务试试：
text
检查当前改动，告诉我主要风险，不要解释背景。
如果是在 Codex CLI 里，可以按项目说明调用：
text
/caveman
它最明显的变化就是回答变短。原来一大段“我将会、接下来、需要注意”，会变成更像任务摘要的东西：改了什么、哪里可能出问题、下一步做什么。
Caveman 把长回答压成结论、风险和下一步
这个工具最适合用同一个问题做前后对比。先不用 Caveman，让 Claude Code 回答一次：
text
检查当前改动，告诉我主要风险，并说明为什么这些风险值得关注。
然后安装 Caveman，再问同一个问题：
bash
claude plugin marketplace add JuliusBrussee/caveman && claude plugin install caveman@caveman
text
检查当前改动，告诉我主要风险，并说明为什么这些风险值得关注。
对比重点不是安装成功，而是同一个问题前后两次回答的差异。好的结果应该是：废话减少，但关键信息还在。
这个工具适合已经熟悉 Claude Code 的人。新手还需要解释，老手才需要压缩。如果你每天跑很多小任务，已经不想看废话，它能省 token，也能省眼睛。

3. Codex plugin for Claude Code：提交前，让另一个模型看一眼

Claude Code 写完代码之后，你可能还想让另一个模型从不同角度看一眼：有没有漏测、有没有边界问题、有没有改出安全风险。
Codex plugin for Claude Code 就是把 Codex 接进 Claude Code。你可以在 Claude Code 里发起 Codex review，也可以把部分任务委托给 Codex。
入口在这里：
  • GitHub：
    https://github.com/openai/codex-plugin-cc
安装前先确认几件事：你需要 ChatGPT 订阅或 OpenAI API Key，需要 Node.js 18.18 或更高版本，最好先确认本机能跑 Codex CLI。
如果还没有 Codex CLI，可以先装：
bash
npm install -g @openai/codex
然后登录：
bash
codex login
Claude Code 里按这几步装插件：
text
/plugin marketplace add openai/codex-plugin-cc
/plugin install codex@openai-codex
/reload-plugins
/codex:setup
装完后，最值得先试的是后台 review：
text
/codex:review --background
/codex:status
/codex:result
/codex:review --background 会把 review 放到后台跑，/codex:status 看进度，/codex:result 取回结果。它适合检查当前未提交改动，也适合在合并前对比主分支做一次审查。
Codex plugin 在合并前给代码做第二次检查
可以先建一个很小的临时 repo，故意制造一个空值风险：
bash
mkdir codex-review-demo
cd codex-review-demo
git init
cat > auth.js <<'JS'
export function getUserName(user) {
  if (!user || !user.profile || !user.profile.name) return "Guest";
  return user.profile.name.trim();
}
JS
git add auth.js
git -c user.name=demo -c user.email=demo@example.com commit -m "add safe fallback"
cat > auth.js <<'JS'
export function getUserName(user) {
  return user.profile.name.trim().toUpperCase();
}
JS
然后在这个目录里打开 Claude Code，跑：
text
/codex:review --background
/codex:status
/codex:result
如果 Codex 抓到 user、profile 或 name 可能为空的问题，这个例子就跑通了。它比安装成功更能说明问题，因为能看到 review 到底在看什么。
更稳的流程是：Claude Code 写代码，Codex plugin 做 review，人来判断哪些建议采纳，然后再跑测试。
不要把第二意见当裁判。Codex 会发现问题，也会误报。它的价值是多一个视角，不是替你做最终判断。

4. Obsidian：给 Claude Code 一个长期 Markdown 工作台

Obsidian 不是 Claude Code 插件，但很适合跟 Claude Code 搭配。
原因很简单,Claude Code 很会写 Markdown，但 Markdown 文件一多，很快会散。项目说明、复盘、提示词、资料整理、发布素材，全在不同文件夹里，模型每次都要重新猜入口。
下载入口：
  • 下载页：
    https://obsidian.md/download
  • 安装帮助：
    https://help.obsidian.md/install
真正关键的是 Vault 路径。比如一台 Mac 上的真实 Obsidian Vault 是：
text
/Users/can/Obsidian
这个文件有几个入口：
text
/Users/can/Obsidian/
├── CLAUDE.md
├── 控制中心/
│   ├── 今日工作台.md
│   ├── 资料入口/
│   ├── 运行结果/
│   └── 备份/
└── Writer/
这里最重要的不是文件夹名字，而是 CLAUDE.md。它相当于给 Claude Code 的 Vault 使用说明。比如里面会写清楚：
  • 先读 控制中心/资料入口/README.md。
  • 不确定资料位置时，先列候选路径，再读取文件。
  • 不要全盘扫描整个 Vault，除非用户明确要求。
  • 当前笔记操作台的输出写入 控制中心/运行结果/当前笔记/。
  • 修改原文前先备份到 控制中心/备份/。
这样 Claude Code 就不会拿到一个大文件夹后乱翻。它会先读规则，再按规则找文件。
实际调用时，可以直接把真实路径给 Claude Code
markdown
请先读取 /Users/can/Obsidian/CLAUDE.md，
再读取 /Users/can/Obsidian/控制中心/今日工作台.md。

不要全盘扫描整个 Vault。
根据 Vault 规则，帮我整理这篇笔记：
/Users/can/Obsidian/Writer/下载别人的 Skill，然后把它们组合成自己的工作流.md

请输出：
1. 这篇文章的核心观点
2. 可以继续扩写的公众号标题
3. 需要补事实核验的地方

结果写入：
/Users/can/Obsidian/控制中心/运行结果/当前笔记/
这条提示词里有三个关键点。
第一，先读规则文件。Claude Code 先知道这个 Vault 怎么组织，而不是凭感觉猜。
第二，给具体笔记路径。比如 Writer/下载别人的 Skill，然后把它们组合成自己的工作流.md，这比说“帮我整理 Obsidian 里的文章”清楚很多。
第三，指定输出目录。这样结果不会散落在聊天窗口里，而是回到 Obsidian 的工作台目录。下次打开 Obsidian，仍然能从 控制中心/运行结果/当前笔记/ 找到。
这样它会先读入口，再按目录规则写入文件。对长期资料来说，这比多装一个插件更有用。
Obsidian 用入口文件收拢分散的 Markdown 资料
Obsidian 的价值就在页面结构。如果是从零开始，可以先在桌面建一个最小 Vault：
bash
mkdir -p ~/Desktop/AI工作流知识库/{工具清单,项目复盘,提示词,发布素材}
cat > ~/Desktop/AI工作流知识库/00-入口.md <<'MD'
# 知识库入口

## 常用路径

- 工具清单：工具清单/
- 项目复盘：项目复盘/
- 提示词：提示词/
- 发布素材：发布素材/

## 使用规则

- 新资料先放收件箱
- 发布过的内容进发布素材
- 每次整理后更新本文件
MD
然后打开 Obsidian，选择 Open folder as vault，选中 ~/Desktop/AI工作流知识库。左侧文件树和 00-入口.md 放在一起看，就能明白它不是“第二大脑”的空话，而是给 Claude Code 一个固定入口。
如果你只有十几个 Markdown 文件，先不用折腾 Obsidian。等资料开始散、找东西开始费劲，再建 Vault。入口文件比花哨配置更重要。
当然，你也可以试试，我的Obsidian的工作台插件，其实相当于内嵌了Claude。

5. NotebookLM 接入：研究资料多了，再接进终端

这里推荐1个 NotebookLM CLI 工具。为了能真的用起来，这里给两个可验证入口。
NotebookLM AI Plugin
入口：
  • GitHub：
    https://github.com/proyecto26/notebooklm-ai-plugin
  • ClaudePluginHub：
    https://www.claudepluginhub.com/plugins/proyecto26-notebooklm-ai-plugin
推荐用 skills 安装：
bash
npx skills add proyecto26/notebooklm-ai-plugin
也可以在 Claude Code 插件系统里安装：
text
/plugin marketplace add proyecto26/notebooklm-ai-plugin
/plugin install notebooklm-ai-plugin
第一次使用需要登录 Google。可以先让 Claude Code 处理登录：
text
Log me into NotebookLM
登录后，再处理具体 notebook：
text
Add this notebook https://notebooklm.google.com/notebook/YOUR_ID and generate a study guide from it.
它的结果不是“聊天更聪明”，而是围绕 NotebookLM 里的资料生成学习指南、幻灯片、音频概览、思维导图、测验、报告等内容。
NotebookLM 接入把网页、PDF 和视频资料变成研究输出
只用 Claude Code，就先试 NotebookLM AI Plugin。
别一上来就折腾 MCP，先确认你真的需要把 NotebookLM 放进终端工作流。

最后怎么选

实际选择时，不建议 5 个一起装。
插件装多了，Claude Code 不一定更强，你的维护成本一定更高。
先选一个最疼的点。装上，用三天。第三天还自然地想用，它才是真的工具。
