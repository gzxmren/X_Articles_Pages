会用 Claude Code 后，真正要补的是 这4 条控制回路
1.1K
先说一个不太客气的判断。
如果你已经用过 Claude Code，还停留在“把 prompt 写清楚一点”，确实会显得小儿科。
因为 Claude Code 用户遇到的混乱，通常不是一句提示词能解决的。
你遇到的更像这些问题。
  • 项目规则写了，但 Claude 该忘还是忘。
  • CLAUDE.md 越写越长，最后变成一锅粥。
  • 每次都让它“先分析再修改”，但一进任务还是直接动手。
  • 你开了权限，弹窗少了，心里却更没底。
  • 你写了 hook，却不知道哪些该自动跑，哪些反而危险。
  • subagent 看起来很高级，但用完只多了一堆摘要，没有让主任务更稳。
这时候再说“给 AI 更多上下文”“让 AI 记住偏好”“写清楚边界”，就太浅了。
真正的问题是，你没有给 Claude Code 建控制回路。
不是更多说明。
是控制回路。
AI 编程工具一旦能读文件、改文件、跑命令、调 MCP、派 subagent，它就不再是聊天框，而是一套会行动的系统。
系统要稳定，不靠一句好 prompt。
靠输入契约、记忆分层、权限边界、验收回路。
这 4 个东西没补上，Claude Code 越能干，你越容易失控。

01. 第一条回路，输入契约

输入契约：先写工作单，再让 agent 动手
很多人用 Claude Code 的方式是这样的。
“帮我看看这个项目。”
“修一下这个 bug。”
“把这个工作流优化一下。”
这不是任务。
这是把一团雾丢给 agent。
Claude Code 很强，它会自己搜文件、读上下文、推断意图、尝试方案。但问题也在这里。你没有定义任务边界时，它会用自己的判断补边界。
补对了，你觉得它聪明。
补错了，你觉得它乱来。
所以第一条回路不是“写更长的 prompt”，而是给每个任务加一个输入契约。
我建议固定 6 项。
这个输入契约，不一定每次都写成大段文字。
你可以把它做成 slash command。
比如 /start-task 先让 Claude Code 输出这 6 项，不允许直接改文件。
它不是命令行脚本，而是 Claude Code 里的项目命令。
你可以建这个文件：
text
.claude/commands/start-task.md
里面写：
markdown
# /start-task

先不要改文件。

请先输出本次任务的 6 项输入契约：

1. 目标
2. 范围
3. 禁区
4. 事实源
5. 完成标准
6. 停止点

如果任一项不清楚，先问我，不要开始执行。
以后你在 Claude Code 里输入：
text
/start-task 修一下 RSS 周报选题工作流
它就会先产出工作单，而不是直接冲进文件里改。
如果任务复杂，第一轮只允许它做三件事。
  1. 读项目入口。
  2. 列出可能相关文件。
  3. 写执行契约。
然后停下。
这一步的价值不是“让 Claude 更听话”。而是把任务从聊天，变成一次可审查的工作单。
用过 Claude Code 的人，真正要升级的第一件事，就是别再让 agent 自己猜任务边界。

02. 第二条回路，记忆分层

记忆分层：把长期规则和一次性背景分开
CLAUDE.md 是好东西。
但很多项目的 CLAUDE.md 最后都会变成垃圾抽屉。
项目结构也写进去。
个人偏好也写进去。
一次性任务背景也写进去。
踩坑复盘也写进去。
发布流程也写进去。
最后 Claude 每次启动都背着一大包东西上路。看起来上下文很足，实际规则互相打架。
Claude Code 的记忆不是越多越好，而是要分层。
我会这样拆。
最关键的是这一句。
CLAUDE.md 不应该是知识库。
它应该是启动配置。
写进 CLAUDE.md 的每一条规则，都应该满足三个条件。
  • 稳定，下周还成立。
  • 可执行，Claude 知道该做什么。
  • 高收益，每次加载都值得占上下文。
如果一条规则只在某个工作流里用，就做 Skill。
如果一条规则只对 docs/ 或 src/api/ 生效，就做 scoped rules。
如果一条规则只是你今天这篇文章的要求，就放当前任务里。
这里也要讲清楚触发方式。
CLAUDE.md 和 rules 是放在项目里，Claude Code 会在会话启动或匹配路径时加载。
Skill 或 slash command 是你在 Claude Code 里输入 /命令名 调用，也可以让 Claude 根据描述在合适的时候使用。
所以不要把所有东西都塞进 CLAUDE.md。一个固定发布流程，更适合变成 /publish-check 这样的 Skill 或 command；一个目录规则，应该放到 rules；只有每次进项目都必须知道的事实，才值得常驻。
这就是第二条回路。
不是让 Claude 记住更多，而是让它在正确的层级记住正确的东西。

03. 第三条回路，权限边界

权限边界：allow、ask、deny 三道门
很多人对 Claude Code 权限的理解太粗。
要么一路默认，每一步都问，最后人被弹窗折磨到麻木。
要么一口气放开，追求“丝滑”，然后开始担心它到底动了什么。
这两种都不是好状态。
权限的目标不是少问。
权限的目标是低风险动作少问，高风险动作必须问，危险动作直接挡。
你可以把动作分成三层。
这里先解释一下这三个词。
你可以把 Claude Code 的权限想成三道门。
allow 是绿灯：这类动作风险低、重复出现、出了问题也容易回滚，所以可以让 Claude Code 直接做。
ask 是黄灯：这类动作不一定危险，但要看时机和上下文，所以每次都应该停下来问你。
deny 是红灯：这类动作不应该发生，不是问不问的问题，而是直接挡住。
比如 git diff 只是看改动，通常可以绿灯。git push 会把东西推到远端，最好黄灯。读取 .env 或跑 rm -rf 这类动作，应该红灯。
这里最容易漏的是 Bash。
Bash 不是一个普通工具。它可以读文件、改文件、联网、删东西、调用二级执行器。
所以 Bash(*) 这种规则，看起来省事，本质上是把方向盘递出去了。
更稳的配置顺序，是先处理最危险的动作，再处理需要确认的动作，最后才放行低风险动作。
先写 deny：哪些事情绝对不能碰。
再写 ask：哪些事情必须停下来问你。
最后才写 allow：哪些低风险动作可以自动放行。
比如一个内容项目，可以允许：
json
{
  "permissions": {
    "allow": [
      "Bash(git diff *)",
      "Bash(python3 工具/app/xiangyu-knowledge-kb-cli/xiangyu-knowledge-kb-cli.py health --brief)"
    ],
    "ask": [
      "Bash(git push *)",
      "Bash(open *)"
    ],
    "deny": [
      "Read(./.env)",
      "Read(./工具/凭证/**)",
      "Bash(rm -rf *)"
    ]
  }
}
这不是模板，别照抄。
重点是分层思路。
能自动的，是你愿意反复承担后果的动作。
必须问的，是需要人判断时机的动作。
直接拒绝的，是不该交给模型试探的动作。
记住，CLAUDE.md 里的“不要读 .env”只是提醒。
permissions.deny 才是边界。

04. 第四条回路，确定性验收

确定性验收：Hook、检查脚本和人工确认
写到这里，很多 Claude Code 用户已经有三层东西了。
  • CLAUDE.md 管规则。
  • permissions 管边界。
  • slash command 或 Skill 管流程。
但还差最后一层。
验收。
没有验收回路，Claude Code 会把“完成了”说得很自然。
问题是它说完成，不等于任务真的完成。
你需要把验收从一句提醒，变成确定性动作。
这里 Hook 就派上用场了。
Hook 的价值不是更强的 prompt。
Hook 是事件发生时一定运行的动作。
常见的硬货用法有 4 种。
如果你的项目是代码项目，最基本的验收回路应该是：
  1. 改文件后自动 format。
  2. 任务结束前跑最小测试。
  3. 输出前检查 diff。
  4. 对高风险改动要求人工确认。
如果你的项目是内容项目，验收也一样可以做。
  1. 写完文章后扫禁用词。
  2. 检查是否有未验证事实。
  3. 检查图片数量和链接。
  4. 检查是否更新索引文件。
Hook 的启用方式也很具体。
它不是你手动输入 /hook 去调用，而是写进项目配置，然后等事件发生时自动触发。
比如你想做到：每次 Claude Code 改完 Markdown，都自动检查禁用词、未验证事实和禁用句式。
你可以在项目里写：
text
.claude/settings.json
加一个 PostToolUse hook：
json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/check-article-words.sh"
          }
        ]
      }
    ]
  }
}
这段配置的意思是：当 Claude Code 用 Edit 或 Write 改完文件后，自动执行 .claude/hooks/check-article-words.sh。
所以 Hook 的价值不是“我想起来就跑一下”。
它的价值是：你不想靠记忆保证它发生。
这才是“AI 工作流”。
不是让 Claude 记得做检查。
是让检查必然发生。

05. Subagent 不是多开一个 AI，而是隔离污染

再说一个很多人容易误用的东西。
Subagent。
很多人看到 subagent，会下意识觉得它是“多一个员工”。于是开始给它起各种角色，研究员、审稿人、架构师、测试员。
这不一定错，但容易变成装饰。
Subagent 最真实的价值，是隔离污染。
有些任务会产生大量中间信息。
  • 搜 200 个文件。
  • 读一堆日志。
  • 查一个 bug 的 5 种假设。
  • 做安全审查。
  • 做性能审查。
  • 对一篇文章挑事实问题。
这些过程如果都塞进主会话，主线很快就脏了。
主会话应该负责目标、判断和最终整合。
Subagent 负责出去翻箱倒柜，最后只把结论带回来。
所以你判断要不要用 subagent，可以问 4 个问题。
如果只是改一个小文件，不要派 subagent。
如果需要你频繁来回确认，也不要派 subagent。
Subagent 不是排场。
它是上下文隔离器。
它也不是 shell 命令。
更常见的入口是 Claude Code 里的：
text
/agents
你可以在里面创建一个项目级 subagent，也可以手写：
text
.claude/agents/article-fact-checker.md
一个最小例子：
markdown
---
name: article-fact-checker
description: 检查文章中的事实、来源、不确定表达和过度承诺。适合在公众号文章初稿完成后使用。
tools: Read, Grep, Glob
model: inherit
---

你是文章事实检查员。

只做四件事：

1. 找出没有来源支撑的事实。
2. 找出被写成确定结论的不确定判断。
3. 找出读者可能看不懂的术语。
4. 输出修改建议，不直接改正文。
然后你可以直接说：
text
请用 article-fact-checker subagent 检查这篇公众号文章。
或者把 description 写清楚，让 Claude 在合适的时候主动调用。

06. 一张升级表

如果你已经用过 Claude Code，可以直接按这张表整理自己的工作流。
这张表，比“写好 prompt”更重要。
因为它把经验从聊天里拿出来，放进系统里。

07. 真正的门槛不是会用，而是会收敛

Claude Code 很容易让人上头。
你看它能读项目、能改文件、能跑命令、能自己找问题，就会忍不住把任务越丢越大。
但真正成熟的用法，反而是收敛。
让常驻规则少一点。
让任务输入硬一点。
让权限边界清一点。
让验收动作确定一点。
让 subagent 只处理会污染主线的子任务。
这时候 Claude Code 才不是一个“聪明但偶尔乱来”的助手。
它开始变成一套可维护的工作流。
会用 Claude Code 的下一步，不是再学 50 个技巧。
是把你的技巧变成系统。
把它用稳，比把它用满重要。
