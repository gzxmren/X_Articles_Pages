OpenClaw 使用一个月，我整理了这十个坑｜小白级避雷攻略
1.9K
说实话，写这篇文章的时候，我还是有点忐忑的。
从 2026 年 2 月 OpenClaw 开始在国内火起来，到现在也一个月了。
我自己从最开始部署，到后面帮几个朋友远程安装，踩过不少坑，也见过太多人在交流群里吐槽。
说实话，这东西真没那么神奇。但用好了，也确实能省你大量时间。
我一直觉得，工具的价值不在于它有多强，而在于你能不能驯服它。
今天把我自己踩过的、还有群里粉丝吐槽最多的十个坑整理出来。希望能帮你少走一些弯路。
坑 1：把 OpenClaw 当搜索引擎用
我自己最开始也犯过这个错。
装好之后第一句话是：「帮我查一下今天天气。」
然后发现回复慢、费用高，还不如直接百度。
后来我才想明白：OpenClaw 的核心能力不是「告诉你怎么做」，而是「直接帮你做完」。
你会发现，这两种用法的效果天差地别：
- ❌ 「帮我找一下今天的AI热点资讯」
- ✅ 「把我找出今天最新的10条AI生图/AI视频热点资讯」
前者它得先搜索、再总结、再回复你。后者它直接执行，一步到位。
记住：它是私人助理，不是百科全书。
坑 2：上来就给满权限
这个坑是做常见的。
刚装好 OpenClaw，把文件读写、Shell 命令、浏览器自动化全开了。结果第二天发现，AI 把工作文件夹重命名了，还自作主张删了几个「看起来没用」的文件。
这就是典型的「给系统太多权力，却不了解它的行为模式」。
OpenClaw 不是人，它不懂你的工作习惯，也不知道哪些文件对你很重要。一开始就给满权限，等于让一个新来的实习生拿着你的管理员密码到处乱点。
那正确的做法是什么？
我自己后来给自己定了一条规矩：
1. 先从只读权限开始
2. 观察执行逻辑，确认行为稳定
3. 逐步开放权限，每次只给一个新能力
4. 重要文件夹设白名单保护
开始使用时我只给它读文件和联网的权限，等我确认它的行为模式稳定了，才开放写入和执行命令的能力。
权限这东西，给出去容易，收回来难。
坑 3：配置没生效
这个坑特别隐蔽。
最开始我在聊天对话窗口中，让 OpenClaw 配置模型和通道以及一些重要的参数。在聊天窗口中说重启生效，实际上并没有。
AI 助理也会说假话。
OpenClaw.json 及重要配置，还是需要在终端重启。指令就一行：
```
openclaw gateway restart
```
10 秒生效，省去半天故障排查。
坑 4：Token 烧得巨多
这个坑看起来低级，但踩的人多到离谱。
做一个Code任务，跑了一晚上线程，第二天账单多了 300 块。
他傻眼了：「我就让它帮我写个代码，怎么这么贵？」
问题就出在模型选择和消费上限上。
我现在的做法是：
1. 主模型用包月 Coding Plan— 阿里云百炼、腾讯云、火山（18000 次/月）
2. 备用模型用国外大模型 — Claude Opus 4.6、GPT-5.4，处理复杂任务
3. 设置消费上限 — Anthropic/OpenAI 控制台都有每日 limit
4. 每周查账单 — 设个周五闹钟，5 分钟搞定
你会发现，这样配置下来，费用至少省 80%。
省下的钱，拿去吃顿好的，不香吗？
坑 5：机器人能发消息但收不到
这是使用飞书机器人的一个坑。
明明机器人在线，但它就是不说话。
最常见的原因：
  • 飞书后台的事件订阅没有配置为「长连接」
  • API 权限没有加够
  • 应用没有发布
去飞书开放平台检查一下。
如果收到配对提示，运行：
openclaw pairing approve feishu 你的配对码
坑 6：没有配置日志记录
很多人装完 OpenClaw 就开始用，从来不看日志。
结果遇到问题的时候，完全不知道哪里出错了。AI 不回消息？不知道。任务执行失败？不知道。费用突然暴涨？还是不知道。
日志是你和 OpenClaw 之间的「黑匣子」。不配置日志，等于蒙着眼睛开车。
正确姿势很简单：
配置文件启用日志，级别设 INFO 或 DEBUG。
遇到问题先看日志，90% 的问题都能找到答案。
日志这东西，平时用不上，用上就能救命。
坑 7：让 AI 助理频繁唤醒
这是最贵的坑，没有之一。
很多人喜欢给 OpenClaw 的心跳设置了一堆没实用的事件，比如1小时查一次邮件、看一下代办。
结果一个月下来，API 费用 200 美元。
问题就出在「唤醒频率」上。
每次唤醒 OpenClaw，它都会调用大模型 API，即使没有新邮件也要「思考」一遍。
那怎么避免？
我自己的做法是：
1. 合理设置唤醒频率，非紧急任务每小时或每天
2. 用 Webhook 触发，别用定时轮询
3. 启用 API 缓存，减少重复调用
4. 定期检查 API 使用，设费用预警
只有真正需要实时响应的任务（比如客服机器人），我才会设置高频唤醒。其他的任务，能用 Webhook 就用 Webhook，能每天跑一次就不要每小时跑。
省下的 200 刀，拿去吃顿好的，不香吗？
坑 8：不给 AI 提供上下文和训练
很多人以为 OpenClaw 装好就能用，结果发现它经常理解错任务，或者执行得很笨拙。
其实 OpenClaw 刚装好的时候，就像一个刚入职的新员工。它不了解你的工作习惯，也不知道你的业务逻辑。你需要「训练」它。
我每次让 OpenClaw 执行新任务之前，都会先给它一个「任务简报」，说明这个任务的背景、目标、注意事项。执行完之后，我会检查结果，如果有问题，就告诉它哪里做错了，下次应该怎么做。
这样训练几次之后，OpenClaw 就能越来越准确地理解你的需求。
AI 不是魔法，它需要你的耐心。
坑 9：不用 Docker 部署
这个坑很多新手会踩。
直接在系统上安装 OpenClaw，看起来简单，但后续维护是个大麻烦。系统更新、依赖冲突、环境变量混乱……任何一个小问题都可能让 OpenClaw 罢工。
更要命的是：如果你想在多台机器上部署，或者想迁移到新服务器，直接安装的方式会让你重新配置一遍所有东西。
我的建议是：
用 Docker 部署。Docker 把所有的依赖、配置、环境都打包在一起，迁移的时候直接搬走整个容器就行。
而且 Docker 还有个好处：隔离性强。OpenClaw 在容器里运行，不会影响你的系统环境，也不会和其他软件冲突。
先用直接安装的方式学习，熟悉了之后，立刻切换到 Docker 部署。
坑 10：不监控 AI 的执行过程
这是最容易被忽视的坑。
很多人设置好 OpenClaw 之后，就完全放手不管了。结果某天突然发现：AI 把重要文件删了，或者执行了一个完全错误的操作。
OpenClaw 很强大，但它不是完美的。它会误判，会出错，会做出你意想不到的操作。如果你不监控它的执行过程，就等于给了它一张无限额度的信用卡，却从来不看账单。
我自己的做法是：
前两周每天晚上都会看一遍 OpenClaw 的执行日志，确认它没有做出奇怪的操作。两周之后，我对它的行为模式有了基本了解，才开始放心让它自动运行。
但即使现在，我也会每周检查一次日志，确保一切正常。
信任，但验证。
最后
潜水一个月，我发现很多人对 OpenClaw 有两种极端的态度：
一种是把它当成万能的魔法，装好就期待它自动解决所有问题；另一种是踩了几个坑之后，就觉得它是个废物，卸载了事。
其实 OpenClaw 就是个工具，一个非常强大的工具。
它能帮你自动化大量重复性工作，能 7×24 小时不间断运行，能执行复杂的多步骤任务。
但它也需要你的配置、训练、监控。
就像你买了一台高性能电脑，不代表你马上就能成为设计大师。你需要学习软件，需要练习技巧，需要不断优化工作流程。
OpenClaw 也是一样。
上面这十个坑，是我见过最多的。如果你能避开这些坑，你的 OpenClaw 使用体验会好 90%。
剩下的 10%，就是你自己的探索了。
每个人的工作场景不同，需求不同，OpenClaw 的用法也不同。但只要你掌握了基本原则——合理授权、及时监控、持续优化——你就能把 OpenClaw 用得越来越顺手。
工具是为人服务的，别反过来。
