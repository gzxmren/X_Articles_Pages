一、基础认知：不写程式的工程师

1.1 破除迷思：到底什么是 n8n 与 Make？为什么这是必修课？
市面上 99% 的营销号，都会把 n8n 和 Make 包装成“小白专用的无代码 (No-code) 神器”，仿佛你只要鼠标点两下，连老奶奶都能做出印钞机。这完全是个美丽的误会。

n8n 和 Make 的真实身份是：“具象化的 API 路由器”与“可视化的逻辑编排引擎”。

想象一下，现在的互联网世界是由无数个“孤岛”组成的。Google 表格是一个孤岛，Telegram 是一个孤岛，ChatGPT 也是一个孤岛。这些孤岛之间本来是互不相通的。程序员每天在敲的代码，有很大一部分的工作，就是在这些孤岛之间“造桥铺路”，让数据可以从 A 岛运到 B 岛。

传统写代码是怎么做的？

你必须面对漆黑的终端机，写下一行行的 Python 或 Node.js，去处理这座桥要用什么材质、数据要装在什么箱子里、遇到断桥怎么办。这些代码全部隐藏在屏幕后面，对技术小白来说，这就是个难以理解的黑盒子。

n8n 与 Make 是怎么做的？

它们直接把这个“黑盒子”砸烂，把里面的齿轮和管线全部摊在阳光下给你看了。你在界面上用鼠标拖拽出来的一个个节点和连线，本质上就是程序员在画“系统架构图”的过程。

那这跟我们接下来要学的 Claude Code 有什么关系？

多数人拿到 Claude Code 这种地表最强的 AI Agent 时，第一个念头都是：“太爽了，帮我写个bot！帮我弄个发文机器人！”然后一按下 Enter，看着终端机喷出一堆红色的报错代码，直接崩溃。

为什么会这样？因为 AI 懂语法，但它不懂你的“业务”。 
[架构具象化] n8n / Make 的底层真实运作逻辑：

  [ 触发点 Trigger ]  ─────>  [ 处理节点 Process ]  ─────>  [ 执行节点 Action ]
  (等待外部事件唤醒)          (数据转换/条件判断)           (发送 API 请求)
   Webhook 叮一声!             提取 JSON 特定字段            夹带 Auth 钥匙出门
1.2 互联网的语言：AI 时代的 6 个核心概念
抛弃维基百科那些不知所云的名词解释，我们用系统串接的视角重新定义：
Webhook (被动触发)：装在系统门口的智能感应门铃。当有人填了表单，对方会主动“叮”一声把数据砸过来唤醒你的系统，而不是写个死循环每秒钟去问“有人吗？”。
HTTP Request (主动出击)：你主动派出的快递员。GET 是去拿数据（比如问天气），POST 是送数据并要求对方执行（比如发消息）。
API (沟通桥梁)：餐厅菜单。你照着菜单给对参数，它就吐给你结果。
JSON (标准物流箱)：数据的公版包装。不管你用 Python 还是 JavaScript，把数据塞进 { "name": "AA" } 这种格式里，全网服务器都能无障碍解析。
Auth (残酷门禁)：API Key 或 OAuth。没有这把钥匙证明你是付费大爷，你的请求永远只能吃 401 拒绝访问的闭门羹。
Deploy (部署上线)：把你的代码从“笔记本里的沙盒”，搬到“云端 24 小时运作的服务器”上。

二、Prompt Engineering：从“讲话”到“下指令”


2.1 为什么模糊需求会毁掉自动化？
“帮我写一个自动化脚本”——新手最最最常见的错误
Claude Code 是一个高执行力的打工人，当你给出模糊边界时，它会自行脑补所有缺失的系统架构。
结果通常是：它用最完美的语法，写出了一套完全无法融入你现有环境、报错连连的废代码。

3.1 环境建置：绝大多数人倒在第一步
市面上过时的教程，开头第一句通常是教你用 npm install -g
@anthropic
-ai/claude-code。
但是依赖全局 npm 安装，不仅极易在不同 Node 版本之间产生冲突，更致命的是权限问题。
当 Claude Code 作为一个 Agent 在你的电脑里游走、尝试修改文件、执行构建脚本时，如果它是用 npm 全局环境跑起来的，经常会遇到 EACCES (权限拒绝) 的报错。
以下是官方推荐、也是目前最稳定无副作用的安装路径：
Mac / Linux 系统
打开你的终端机 (Terminal 或 iTerm2)，直接贴上这行脚本：
bash
curl -fsSL https://claude.ai/install.sh | bash
接下来选择登录方式，用浏览器授权：用键盘方向键，选中「1. Claude account with subscription」，回车。浏览器会弹出来，点击黑色的「Authorize」按钮
终端里出现「Welcome back!」或者 Claude 的交互界面，说明安装成功！
Windows 系统
打开 PowerShell，执行以下指令：
powershell
irm https://claude.ai/install.ps1 | iex
看到进度条或滚动文字，等待期间不要关闭窗口
输入 claude，回车Enter，用浏览器登录：
和 Mac 步骤一样：输入 claude 回车，浏览器弹出，点「Authorize」，完成
  • 前置排障：如果系统跳出红字阻挡，那是微软的基础防御机制 (Execution Policy)。先执行 Set-ExecutionPolicy RemoteSigned -Scope CurrentUser 放行，再重新安装即可。
3.2 项目大脑：CLAUDE.md 与 /compact 内存回收机制
上下文窗口（Context Window）是你最昂贵的资源
不要把 CLAUDE.md 当作许愿池
Claude Code 在启动时，会主动去寻找项目根目录下的 CLAUDE.md 文件。很多小白以为这是用来写需求的，洋洋洒洒写了三千字的功能规划。 大错特错。 CLAUDE.md 是这个项目的“最高宪法”与“系统提示词”。它会在你的每一次对话中，被强制塞进上下文里。你写得越长，AI 每次思考的负担就越重，越容易引发幻觉。
一份完美的 CLAUDE.md 应该极度克制，只包含以下三件事：
  1. 架构约定：例如 Strictly use TypeScript. No 'any' types allowed.
  2. 禁止事项：例如 Do NOT modify database schema files under /prisma without explicit confirmation.
  3. 构建指令：例如 Test command: pnpm run test:watch

PART 2: 分层治理策略 (防污染护城河)

- 始终常驻层：CLAUDE.md
  [用途] 仅放全局契约、构建命令、绝对禁止事项。原则：越短越好。

- 按路径加载：.clauderules
  [用途] 特定目录或语言的专属规则。避免让 AI 在写前端 React 时，脑子里还载入后端的 Python 规范。

- 按需加载层：Skills
  [用途] 特定工作流或垂直领域知识。随插即用，用完即弃。

- 隔离运行层：Subagents (子代理)
  [用途] 遇到需要翻阅大量长文档、或者并行查错时开启。让它在独立沙箱里跑，防止繁杂的日志污染主线对话。

- 零消耗拦截：Hooks
  [用途] 确定性的阻断脚本。根本不进入上下文，直接在系统底层进行物理拦截与审计。
内存回收器：/compact
当你在终端机里跟 Claude 查 Bug，查了十几轮，屏幕上堆满了各种红字报错和长篇大论的代码分析。此时，AI 的上下文窗口已经被这些“垃圾日志”严重污染，它的回复开始鬼打墙，甚至忘记了你最初要修的是什么。
遇到这种情况，不要死磕。 输入指令：/compact
[系统底层执行逻辑]
1. 侦测到 /compact 指令
2. Claude 暂停当前任务
3. 自动总结先前的所有对话、报错、已排除的原因
4. 将原本庞大的对话树，压缩成一份极简的“进度报告”
5. 清空旧上下文，带着进度报告重新开局
3.3 Auto Mode：平衡效率与安全的分类器机制
传统 Agent 有一个极其反人类的设计：无限弹窗。 AI 决定建立一个文件，系统问你 Approve? (Y/n)；AI 决定跑个测试，系统问你 Approve? (Y/n)。你以为你在自动化，实际上你变成了无情的敲 Y 机器。
Anthropic 在 2026 年 3 月推出的 Auto Mode 彻底解决了这个痛点。 不要害怕开启它。Auto Mode 的底层并没有完全放权，而是内置了一套极为聪明的意图分类器 (Classifiers)。
  • 自动放行 (Green Zone)：读取文件目录、执行 npm run lint、修改常规业务逻辑代码、运行本地测试。这些它会默默秒速完成。
  • 强制拦截 (Red Zone)：当 AI 试图执行 rm -rf 批量删除、修改全局系统变量、覆盖核心环境配置文件 (.env)、或者执行危险的数据库 Drop 动作时，分类器会瞬间阻断执行链，并将控制权交还给你进行人工审批。
3.4 Hooks：系统级别的物理拦截
如果说 CLAUDE.md 是用法律（文字）来约束 AI，那 Hooks 就是直接在门上加了一把物理大锁。
有时候，你在文字里苦苦哀求它“千万不要用 npm 安装套件，我们团队只用 pnpm”，但它遇到某个顽固报错时，一着急可能还是会顺手打出 npm install。
可以利用 PreToolUse Hook 来掌控绝对的执行流。

四、实战案例


4.1使用场景
哪怕你一行代码都不写，只要你掌握了“明确上下文与验收标准”的原则，Claude Code 就是你电脑里最可怕的效率怪兽。
以下是几个非编程使用场景。打开你的终端机，启动 Claude Code，这些指令你可以直接复制粘贴给它：
# 1. 自媒体 / 内容创作（内容缺口分析）
"读取当前目录下近 3 个月的所有推文草稿 (.md 文件)，帮我按核心话题进行分类整理。找出哪些行业视角的切入点是我还没写过的，最后生成一份包含 5 个新角度的选题建议报告，存为 next_topics.md"

# 2. 销售 / 客户管理（非结构化数据转结构化）
"把 /chats 目录下的这 6 个月客户聊天记录文本扫一遍。提取每个客户的核心需求和痛点，帮我生成一份结构化的 JSON 客户画像报告，包含字段：客户名、预算级别、核心痛点、建议跟进策略。"

# 3. 知识 / 桌面文件管理（本地文件系统操作）
"扫描我的 /Desktop 目录，这里有大约 300 个混乱的文件。请按工作、学习、生活建三个子文件夹并分类移入；把命名混乱的文件用统一格式重命名；最后，把修改日期在一年以上且未打开的文件列出一个清单给我，让我决定是否批量删除。"

# 4. 数据分析（表格处理与洞察）
"读取这个 data.csv 互动数据表，帮我计算过去 6 个月哪一类内容的完播率和互动率最高，总结出 3 条底层规律，并把结论写进一个 markdown 分析报告里。"

4.3 混合语言项目的 Hooks 实践
当项目变得复杂，比如你接手了一个底层是 Rust、上面跑着 Lua 脚本的混合语言项目，你就会发现一个残酷的现实：环境如果不透明，AI 就只能靠猜，一猜就会失控崩溃。
別让 AI 自己去瞎测，而是利用 Hooks 进行“按文件类型触发的语法检查”。这是我实际在用的 .claude/settings.json 配置，它能在 AI 每次修改完文件后，立刻物理触发编译检查：
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit",
        "pattern": "*.rs",
        "hooks": [
          {
            "type": "command",
            "command": "cargo check 2>&1 | head -30",
            "statusMessage": "Checking Rust..."
          }
        ]
      },
      {
        "matcher": "Edit",
        "pattern": "*.lua",
        "hooks": [
          {
            "type": "command",
            "command": "luajit -b $FILE /dev/null 2>&1 | head -10",
            "statusMessage": "Checking Lua syntax..."
          }
        ]
      }
    ]
  }
}
4.4 结语：不可逾越的黄金法则与反思
用 Claude Code 折腾大半年下来，会发现所有人都会不可避免地经历三个阶段：
  1. 玩具阶段：这个功能怎么用？指令怎么敲？（沉迷于跟它对话，把它当高级chatgpt/gemini）
  2. 助理阶段：怎么配置环境？怎么让它少报错？（开始研究 Prompt 和上下文管理）
  3. 架构师阶段：怎么设定系统边界？怎么让 Agent 在约束下自己跑完完整的闭环？
到了第三阶段，你的关注点会发生根本性的转移。你不再关心某行具体的代码该怎么写，你关心的是整个数据流的稳定性、内存的释放时机，以及系统权限的物理拦截。
最后
如果一个任务，你连“什么叫做完（验收标准）”都说不清楚，那就大概率不适合直接扔给 Claude 去自主完成。
没有验证标准，AI 再聪明也跑不出正确的答案。建立边界，设定验收标准，然后，放开手让它去接管你的电脑吧。
