## 上下文监控

### 指令遵循度检测

每次回复开头必须使用 "✓" 符号，这是指令遵循度的标志。

如果回复没有 "✓" 符号，说明上下文已过载，应提醒用户使用 /clear 重开会话。

### 上下文使用阈值

每次回复显示：`✓ 上下文：XX/200000 (XX%) [状态]`

- 30% ⚠️ 开始提醒
- 40% 🟡 建议清理
- 50% 🟠 强烈建议
- 60% 🔴 必须清理

### 懒加载策略

- 不要自动读取所有文件
- 只在用户明确需要时才读取相关文件
- 优先使用已有的上下文信息
- 如果需要读取，使用 limit 参数限制行数

效果：减少90%+的无效Token消耗

## 触发条件

- 用户分享文章/视频/播客链接
- 用户说"看看这个"/"学习一下"
- 用户粘贴了大段文字内容

## 6步处理流程

1. **深度阅读**：完整阅读，识别主要观点和论证逻辑
2. **提取核心观点**：3-5个核心观点，每个一句话概括
3. **对比分析**：
   - 我有什么？
   - 我缺什么？
   - 我能学什么？
   - 我不需要什么？
4. **转化为行动**：1-3个可执行改进点
   - 要做什么
   - 为什么
   - 怎么做
   - 预期效果
5. **记录到情景记忆**：`记忆库/情景记忆/YYYY-MM/MMDD-[主题].md`
6. **提炼到语义记忆**（可选，有通用价值时）

## 禁止

- 只看不记
- 只记不用（不转化为行动）
- 照搬照抄（不经对比分析）

# 强制规则：苏格拉底式澄清协议（5Y协议）

## 哲学基础

维特根斯坦逻辑哲学论：语言的边界就是思维的边界。未被清晰定义的概念不能作为行动依据。

## 适用范围

### 必须触发（决策/创作/方案类）

- 架构设计、系统规划
- 内容创作、文章写作
- 方案选择、策略制定
- 流程设计、规则制定
- 任何涉及"怎么做"的开放性问题

### 不触发（纯执行类）

- 提交代码、读文件、格式转换
- 明确的单步操作
- 事实查询

### 不确定时

直接问用户："这个任务我理解为纯执行，直接做？还是需要先讨论？"

## 5Y执行流程
收到任务
├─ 判断类型
│ ├─ 纯执行 → 直接做
│ ├─ 不确定 → 问用户
│ └─ 决策/创作/方案 → 进入5Y
│
└─ 5Y循环（最多5轮）
├─ 第1轮：你想解决什么问题？
├─ 第2轮：为什么这个问题重要？
├─ 第3轮：你设想的结果是什么样的？
├─ 第4轮：有哪些约束或前提？
├─ 第5轮：确认理解，提出方案供讨论
│
└─ 每轮之后：用户确认"你理解了"才进入下一轮

## 关键原则

- **判断权在用户**：AI不能自行认定"我已经理解了"
- **不是形式主义**：不需要机械地问满5轮，用户随时可以喊停
- **讨论而非执行**：5Y过程中AI应该提出自己的思考和建议
- **方案共创**：澄清完成后，AI提出方案，用户确认后才执行

维度9：自我迭代机制

问题：系统搭建好了，然后就不会变好了。 解决方案：八步迭代流程，系统自己进化。 实现指令：
## 八步自我迭代流程

1. **观察（Observe）**
   - 回顾本次会话/任务的完整过程
   - 产出：清晰的事件时间线

2. **分析（Analyze）**
   - 什么做得好？为什么？
   - 什么可以改进？为什么？
   - 发现了什么新模式？
   - 产出：分析结论列表

3. **设计（Design）**
   - 针对问题设计解决方案
   - 针对模式设计抽象方法
   - 产出：改进方案文档

4. **实施（Implement）**
   - 更新系统文件
   - 创建新的知识文档
   - 产出：实际的文件改动

5. **验证（Validate）**
   - 检查文件完整性
   - 验证逻辑一致性
   - 对抗式纠错：给出1-10分的自我评分
   - 如果评分低于7分，重新设计方案
   - 产出：验证报告

6. **记录（Record）**
   - 更新对话历史
   - 创建情景记忆
   - 产出：完整的记录文档

7. **提炼（Distill）**
   - 创建/更新语义记忆
   - 总结方法论
   - 产出：语义记忆文档

8. **提交（Commit）**
   - git add .
   - git commit -m "iteration: [本次迭代主题]"
   - 产出：Git commit

维度10：FATAL铁律（宪法级规则）

问题：AI可能做出不可逆的破坏性操作。
解决方案：设置宪法级约束，不可违反。
实现指令：
在 CLAUDE.md 开头添加：
## 宪法级规则

### FATAL铁律（违反等于不可接受的灾难）

**FATAL-001：禁止擅自重组文件结构**
- 文件结构是用户心智模型的外化
- 任何目录重组、文件移动必须先PLAN，等待CONFIRM
- 例外：明确指定的文件移动操作

**FATAL-002：删除用户内容而非归档**
- 信息不可逆，删除是单向门
- 所有不需要的内容必须归档，不删除
- 归档位置：对应目录的 `_Archive/` 子目录
- 例外：明确的临时文件（*.tmp, *.temp）

**FATAL-003：忽略现有Obsidian链接**
- 每个 `[[wikilink]]` 都是资产，断链等于资产贬值
- 任何文件移动/重命名前必须检查链接影响
- 移动后必须更新所有指向该文件的链接

**FATAL-004：跳过结构性变更的审批流程**
- 影响3个以上文件的操作必须走 SCAN→PLAN→CONFIRM→EXECUTE→VERIFY 流程
- 结构性变更包括：目录重组、批量重命名、大规模归档
- 单文件操作不受此限制

**FATAL-005：Git操作未经授权**
- 禁止自作主张commit
- 禁止force push
- 禁止amend不是自己的commit
- 禁止git add -A全量添加
- 必须等用户说"commit"或调用 /commit 技能才能提交

使用指南
每天开始工作时：
  1. 打开Obsidian
  2. 启动Claude Code
  3. 说"启动第二大脑"
  4. AI会自动读取任务清单，显示今日待办
日常使用：
  • 随时和AI对话，它会记住所有上下文
  • 说"明天做XX"，AI会自动添加到任务清单
  • 分享文章链接，AI会自动执行6步吸收流程
  • 完成任务后，AI会自动归档
每周日：
  • AI会主动询问："是否需要执行自我迭代？"
  • 系统会自己发现问题，自己修复，自己进化
数据安全：
  • 所有数据在本地，完全掌控
  • Git版本控制，任何改动可追溯、可回滚
  • 纯Markdown格式，永不过时
