Part 5 of 12 - Vectorized Backtesting: Testing Strategies on Real Data

In the previous article, we implemented three classic strategies: SMA Crossover, momentum, and mean reversion. For each one, we wrote the code, generated signals, and visualized them on the BTC chart. But signals alone are not results. We still do not know whether any of these strategies would have actually made money.
In this article, we find out. We will run each strategy through historical BTC data, calculate real returns, maximum drawdown, and Sharpe ratio - then compare all three strategies against each other and against a simple buy-and-hold.

What Is Backtesting

Backtesting means testing a trading strategy on historical data. The logic is simple: if we had applied this strategy in the past, what would have happened?
It sounds obvious, but there is an important caveat: a backtest does not guarantee the strategy will work in the future. Markets change, patterns disappear, trading conditions differ. A good backtest is a necessary - but not sufficient - condition for deploying a strategy in the real world.
That said, backtesting is the first mandatory filter. If a strategy does not work on historical data, it almost certainly will not work on a live market either.
  • Two Approaches: Vectorized and Event-Driven
There are two main ways to implement a backtest:
Vectorized backtesting - we apply mathematical operations to the entire data array at once. Fast, concise, and convenient for research. This is the approach we use in this article.
Event-driven backtesting - we process data one bar at a time, as if time is moving forward. Slower, but more accurate: it can account for commissions, slippage, and partial order fills. That is the topic of Article 6.
For a first pass at testing a hypothesis, the vectorized approach works perfectly.

How Vectorized Backtesting Works

The idea is simple. We have two data series:
- `log_return` - the logarithmic daily return of BTC - `position` - the strategy's position on each day (1 = in the market, 0 = out of the market)
The strategy's return on each day is just the product of the two:

strategy_return[t] = log_return[t] * position[t]
If the position is 1, we get the market return. If 0, we get zero - we are out of the market. That is the entire vectorized backtest in one line.
Cumulative return is calculated by taking the running sum of log returns:
python
cumulative = strategy_returns.cumsum().apply(np.exp)
This is where the beauty of log returns shines: you can simply add them up

The Backtest Function

Let us write a universal function that takes a strategy DataFrame and returns all the metrics we need:
python
def backtest(df, strategy_name="Strategy"):
    """
    Vectorized backtest of a strategy.

    Parameters:
        df            -- DataFrame with 'log_return' and 'position' columns
        strategy_name -- strategy name (for display)

    Returns:
        dict of metrics and a DataFrame with results
    """
    result = df.copy().dropna(subset=["position", "log_return"])

    # Strategy return: market return multiplied by position
    result["strategy_return"] = result["log_return"] * result["position"]

    # Cumulative return: strategy vs buy-and-hold
    result["cumret_strategy"]   = result["strategy_return"].cumsum().apply(np.exp)
    result["cumret_buyandhold"] = result["log_return"].cumsum().apply(np.exp)

    # Metrics
    total_return  = result["cumret_strategy"].iloc[-1] - 1
    bh_return     = result["cumret_buyandhold"].iloc[-1] - 1

    annual_return = (1 + total_return) ** (252 / len(result)) - 1
    annual_vol    = result["strategy_return"].std() * np.sqrt(252)
    sharpe        = annual_return / annual_vol if annual_vol > 0 else 0

    # Maximum drawdown
    rolling_max  = result["cumret_strategy"].cummax()
    drawdown     = (result["cumret_strategy"] - rolling_max) / rolling_max
    max_drawdown = drawdown.min()

    # Win rate: fraction of days the strategy made money
    winning_days = (result["strategy_return"] > 0).sum()
    trading_days = (result["position"] == 1).sum()
    win_rate     = winning_days / trading_days if trading_days > 0 else 0

    metrics = {
        "name":          strategy_name,
        "total_return":  total_return,
        "bh_return":     bh_return,
        "annual_return": annual_return,
        "annual_vol":    annual_vol,
        "sharpe":        sharpe,
        "max_drawdown":  max_drawdown,
        "win_rate":      win_rate,
        "trading_days":  int(trading_days),
    }

    return metrics, result

Metrics: What They Actually Mean

Before interpreting results, let us make sure we understand each metric.
  • Total Return
The simplest metric: how much the capital would have grown over the entire period. `+62.4%` means $1,000 would have become $1,624.
But comparing strategies by return alone is not enough. A strategy that returned 100% while losing 80% along the way is far worse than one that returned 40% with no serious drawdowns.
  • Sharpe Ratio
The Sharpe ratio is the primary metric for comparing strategies. It measures return per unit of risk:
``` Sharpe = Annual Return / Annual Volatility ```
Rough benchmarks: - `Sharpe < 0.5` - weak strategy - `0.5 -- 1.0` - acceptable - `1.0 -- 2.0` - good - `> 2.0` - excellent (or a bug in the backtest)
We are using a simplified version without the risk-free rate. In real systems, the return on a risk-free asset (e.g., short-term US Treasury bills) is typically subtracted from the numerator.
  • Maximum Drawdown
Maximum Drawdown is the largest peak-to-trough decline in portfolio value:
python
rolling_max = cumret.cummax()
drawdown    = (cumret - rolling_max) / rolling_max
max_dd      = drawdown.min()
If Sharpe measures average performance, drawdown captures the worst-case scenario. A strategy with a -50% drawdown needs a +100% recovery just to break even. That is both psychologically brutal and practically dangerous.
  • Win Rate
The fraction of days the strategy made money, out of days it was in the market. Intuitively, a high win rate sounds good - but it does not have to be. A strategy can win small amounts often while losing large amounts rarely. What matters is not the frequency of wins, but the ratio of the average win to the average loss.

def plot_equity_curves(results: dict):
    """
    results -- dict of {"Name": result_DataFrame}
    """
    fig, axes = plt.subplots(2, 1, figsize=(14, 9),
                              gridspec_kw={"height_ratios": [3, 1]})

    colors = ["steelblue", "darkorange", "forestgreen"]

    first_res = list(results.values())[0]
    axes[0].plot(first_res.index, first_res["cumret_buyandhold"],
                 linewidth=1.2, color="black", linestyle="--",
                 label="Buy & Hold", alpha=0.7)

    for (name, res), color in zip(results.items(), colors):
        axes[0].plot(res.index, res["cumret_strategy"],
                     linewidth=1.2, color=color, label=name)

    axes[0].set_title("Equity Curve: Strategies vs Buy & Hold (BTC/USDT)")
    axes[0].set_ylabel("Cumulative Return (start = 1)")
    axes[0].legend()
    axes[0].grid(True, alpha=0.3)
    axes[0].axhline(1, color="gray", linewidth=0.6, linestyle=":")

    for (name, res), color in zip(results.items(), colors):
        rolling_max = res["cumret_strategy"].cummax()
        drawdown    = (res["cumret_strategy"] - rolling_max) / rolling_max
        axes[1].fill_between(res.index, 0, drawdown * 100,
                              alpha=0.4, color=color, label=name)

    axes[1].set_ylabel("Drawdown (%)")
    axes[1].set_xlabel("Date")
    axes[1].legend(loc="lower left", fontsize=8)
    axes[1].grid(True, alpha=0.3)

    plt.tight_layout()
    plt.show()

Interpreting the Results

Let us look at the numbers honestly.
  • SMA Crossover shows the best Sharpe ratio among the three. It is a trend-following approach, and BTC is a trending asset - which explains the relatively decent results. The main problem: in a sideways market, the strategy generates plenty of false signals and bleeds money on commissions.

  • Momentum behaves similarly to SMA but reacts faster. More trades means more transaction costs in real trading. Frequent entries and exits also mean the strategy may get shaken out of a position right before a reversal.

  • Mean Reversion lags on return but has the lowest volatility and the smallest drawdown of the three. The problem is that BTC is not an ideal mean-reversion asset: it is prone to strong trends where the price can diverge from its mean far beyond what a z-score of -1 would suggest.

  • Buy-and-hold on BTC generally outperforms all three strategies on return during a bull market - but it carries the largest drawdowns, sometimes -70% or worse. This is our honest benchmark: the goal is not just to beat the market on return, but to do it with a better risk/return profile.
This picture is typical of most simple strategies: they do not consistently beat the market, but they can offer a different risk profile. That is already valuable.

Pitfalls to Avoid

Backtesting is a minefield. Here are the most common mistakes.
  • Look-Ahead Bias
The most dangerous error. If your strategy uses data that would not have been available at the moment of decision-making, the backtest will show unrealistically good results.
We already protected ourselves against this in Article 4: the position is always shifted one day forward (`shift(1)`). But be careful - when working with rolling indicators, machine learning, or any calculation that touches future data, this mistake is easy to make again.
  • Survivorship Bias
If you test a strategy only on assets that exist today, you automatically exclude everything that went bankrupt or stopped trading. BTC is a safe choice in this regard - it is still here. But when testing across a basket of assets (say, the top 100 cryptocurrencies), this needs to be taken into account.
  • Overfitting
If you spend a long time tuning strategy parameters to fit a specific historical period, you will end up with a strategy that works perfectly on that data and falls apart on new data. In Article 6, we will cover walk-forward analysis - a methodology that helps address this.
  • Transaction Costs
The vectorized backtest in its current form does not account for commissions or spreads. On Binance, the fee is 0.1% per side (or less with BNB). For strategies with infrequent trades this is not critical, but for momentum with frequent switches it adds up. We will add transaction costs in Article 6.
